# Routing Rules

Use this reference before selecting any downstream skill, changing `current_stage`, or applying module maturity changes.

1. Start by ensuring workflow state is initialized for the current run. If runtime persistence is enabled, initialize the runtime artifact before any stage routing.
1.1. Unless the user explicitly overrides the language for a specific artifact, every workflow artifact produced during routing must use Simplified Chinese for its human-readable content, including PRD text, RD text, `DESIGN.md`, workflow records, checklists, receipts, review notes, and visible prototype copy.
2. Before deciding or invoking anything downstream, derive one route lock and run the preflight gate. If either step fails, stop and record a blocker instead of letting the downstream skill reinterpret the workflow.
3. If `confirmation_status` is `pending_confirmation`, do not switch to the next process and do not apply queued status changes, unless `--auto` or `--full-auto` is active. In `--auto` mode, ordinary downstream confirmation gates are auto-confirmed until workflow completion or a blocker is reached. In `--full-auto`, deterministic human-facing gates may also be auto-confirmed when the control contracts prove there is exactly one supported default.
4. If the user explicitly confirms a pending transition or status update, promote the queued values, clear pending fields, and continue normal routing from the newly confirmed state.
5. If the user rejects a pending transition or status update, keep the current confirmed state, write the rejection reason, and route back to the skill that must revise artifacts.
6. If a specialist skill returns `blocked`, `not_executed`, or a receipt that fails route-lock validation, do not advance. Keep `current_stage` on the last confirmed stage, clear queued transitions, and record the blocker or route-drift reason.
7. If the input is still a rough idea, reference bundle, or abstract request that is not yet concrete enough to discuss through pages, set or keep `current_stage=idea_sketch_brainstorming`, read `idea-sketch-flow.md`, brainstorm the product direction, and generate `1-3` key-page sketches before any PRD work begins.
8. If `idea_sketch_brainstorming` has not yet produced confirmed key-page sketches, do not call `requirements-prd-flow.md`, `flutter-prd-rd-writer`, technical baseline generation, Product Design brief confirmation, shared freeze, architecture, or implementation planning. Record `required_inputs=idea_sketch_confirmation` and stop.
9. Before leaving `idea_sketch_brainstorming`, ensure the user has confirmed product direction, page structure, and broad visual style, and ensure `docs/project/idea-sketch-brief.md` exists on disk.
10. If the idea-sketch confirmation has passed and `docs/project/idea-sketch-brief.md` exists, queue `pending_next_stage=requirements_brainstorming` until confirmation unless `--auto` or `--full-auto` is active.
11. If the input is already concrete enough for PRD work but no PRD artifact exists yet, or if the workflow has just confirmed `docs/project/idea-sketch-brief.md`, set or keep `current_stage=requirements_brainstorming`, read `requirements-prd-flow.md`, brainstorm the requirement space, resolve all decision-blocking questions, and generate a PRD before advancing.
12. If requirements brainstorming finds unresolved decision-blocking questions that cannot be answered from context or by low-risk defaults, do not call `flutter-prd-rd-writer`. Record the question ledger and stop with `required_inputs`.
13. Before promoting the run to `prd_ready`, read `prd-template.md` and `prd-completeness-gate.md`, ensure the PRD artifact exists on disk, ensure the question ledger is embedded or linked, and ensure the completeness outcome is `ready_for_prd_ready`.
13.1. If the PRD artifact is not written in Simplified Chinese and the user did not explicitly request another language, do not promote it to `prd_ready`.
14. If all PRD-blocking questions are resolved or explicitly defaulted and the completeness gate passes, queue `pending_next_stage=prd_ready` until confirmation unless `--auto` or `--full-auto` is active.
15. If the input is already a PRD, broad RD, or feature brief, use `flutter-prd-rd-writer` first to derive or refresh the global technical baseline, then queue or promote `technical_baseline_ready`.
16. If a global technical baseline exists but the target design-device preset and base resolution are not yet frozen for this design cycle, keep `current_stage=technical_baseline_ready`, record `required_inputs=design_device_preset_and_resolution`, and stop. In manual mode, offer at least these common iPhone presets unless the user already chose another viewport: `375 x 667 px`, `390 x 844 px`, `393 x 852 px`, and `430 x 932 px`. In `--auto` or `--full-auto`, if no frozen viewport exists yet, auto-freeze `390 x 844 px` as the design viewport and continue.
17. If a global technical baseline exists but the workflow lacks a confirmed Product Design brief for this request, first read `design-quality-guidance.md`, then route to `@product-design` `get-context`. Product design must be briefed before final product direction confirmation.
17.1. Use `@product-design` `get-context` only for design-brief confirmation and design-side clarification. Do not treat it as a substitute for PRD authoring, technical baseline generation, or implementation planning.
18. If a global technical baseline exists but the workflow lacks a confirmed structured Product Design clarification packet for later module splitting, keep the run on the shared-direction branch and route to `@product-design` scope clarification before any shared freeze completion or module splitting. That clarification must lock core user journeys, page families, critical states, interaction goals, explicit `platform_identifier`, platform-aware shell or navigation expectations, and per-surface information-density posture.
18.1. If the workflow still lacks strong evidence about user pain, current UX friction, or screenshot-level product issues, it may route to `@product-design` `research` or `audit` before shared direction work. Use `research` for source-grounded user-problem synthesis and `audit` for screenshot-based flow critique. Do not use either one to bypass PRD completeness or freeze gates.
19. Do not generate module `impl.md` documents, module implementation-stage effect images, or module freeze artifacts before the shared/global design freeze has completed.
20. If the common public shell is not yet explicitly agreed with the user, set or keep `current_stage=shared_design_direction`, record `required_inputs=public_shell_confirmation`, and stop. In `--full-auto`, if the current shell packet has exactly one supported shared-shell contract and no competing shell option remains in scope, auto-confirm it and continue instead of stopping.
21. If the Product Design brief for this request has not yet been confirmed, set or keep `current_stage=shared_design_direction`, route to `@product-design` `get-context`, and stop until the brief is confirmed.
22. If the Product Design brief is confirmed but the workflow has not yet completed a Mobbin-backed inspiration pass for the current design cycle, keep `current_stage=shared_design_direction`, route to `@product-design` through a Mobbin-first shared/global style recommendation pass, and stop any direction recommendation, representative-sketch generation, final effect-image generation, or downstream freeze work until that pass is complete.
22.0.0. Do not open commercial design exploration until `mandatory_mobbin_reference.status=complete`. That receipt must include relevant reference screen evidence, extracted layout and hierarchy decisions, and why those references fit the product's target platform and viewport.
22.0.0.1. Do not lock the selected commercial direction, shared native HTML prototype, representative sketch, final effect image, Pencil source, or shared freeze unless `fixed_style_direction.status=selected` and `strong_hierarchy_contract.status=recorded`. The fixed style direction is the single style source for downstream work, and the strong hierarchy contract must name the hierarchy contrast ladder for primary action, primary content, secondary content, metadata, support copy, and background surfaces.
22.0.0.2. Do not generate global page design drafts. The shared/global design route must output `global_style_scheme.status=selected` with `theme_and_style_only=true`, `no_global_page_design_draft=true`, and `page_design_deferred_to_module_stage=true`. Global design may define theme tokens, palette posture, typography posture, density posture, surface depth, component family, CTA posture, icon or image treatment, hierarchy rules, motion restraint, copy posture, public-shell principles, and reusable component rules, and it may include one `global_style_experience_image` with `non_page_design_evidence=true` to make the selected style direction visually reviewable. That image must remain a non-page style board and must not create page-level HTML prototypes, representative page sketches, final effect images, atlas bundles, Pencil page designs, shared page comps, or hidden page-layout sources. Mobbin references are mandatory style evidence only, not page-design sources.
22.0. After the Product Design brief and Mobbin-backed inspiration pass are available, keep `commercial_design_exploration.direction_space_open=true` until one selected commercial direction is recorded. Do not lock the shared native HTML prototype on any legacy branch before this exploration records whether first-screen structure, module order, CTA placement, navigation emphasis, information density, copy compression, and component rhythm should be recomposed for commercial product quality; on the current global style-only route, record those decisions as style-system constraints instead of page layout drafts.
22.1. Before shared/global freeze may begin, route the shared/global scope to `Product Design:ideate` and a `global_style_scheme` selection step rather than a page prototype step. The scheme must select one fixed commercial style direction, may generate `global_style_experience_image` through `Product Design:ideate`, and must record the Mobbin-backed evidence, strong hierarchy contract, minimal-copy contract, public-shell posture, component-family posture, and theme-token intent.
22.2. In manual mode, once the `global_style_scheme` is reviewable, keep `current_stage` on the last confirmed stage, queue `pending_next_stage=product_direction_confirmed`, record the style-scheme receipt, and stop for explicit user confirmation. Do not start shared/global representative sketch generation, final effect-image generation, atlas analysis, shared Pencil work, or page-design draft generation before or after that confirmation.
22.2.1. Do not accept a shared native HTML prototype as a global-stage substitute unless a legacy branch has already produced it and its receipt includes `commercial_surface_gate.result=passed`; even then, it may only inform the `global_style_scheme` and must not become a required global page design draft. If the gate returns `revision_required`, keep `current_stage` on the last confirmed stage and revise the theme/style scheme rather than continuing into page generation.
22.3. If review of the global style scheme changes wording posture, module grouping principles, action priority, public-shell posture, or component rhythm, treat that outcome as a requirements change. Update the affected PRD sections, `DESIGN.md`, shared design packet notes, and workflow record before shared freeze routing.
22.4. After the global style scheme is confirmed, treat it as the first-priority source for theme, hierarchy, density, component family, action emphasis, and copy posture. Later module HTML prototypes, module effect images, and module Pencil designs may apply those constraints to concrete pages, but they must not replace the global style source.
22.5. After the exploration window closes, set `commercial_design_exploration.freeze_boundary=global_style_scheme_selected`. Later module sketches, module effect images, Pencil design, and Flutter restoration may refine page fidelity, but they must not reopen broad structure recomposition unless the user explicitly restarts the design cycle.
23. If the Product Design brief is confirmed, the Mobbin-backed inspiration pass is complete, and no selected `global_style_scheme` exists yet, keep `current_stage=shared_design_direction`, route to `@product-design` visual ideation using the approved reference images plus a Mobbin-backed shared/global style recommendation pass, then record exactly one selected style scheme under the confirmed Product Design brief and commercial product UI/UX constraints. Treat `https://mobbin.com/` only as inspiration evidence for this pass, not as the formal downstream design source. Do not generate page-level sketches, page comps, final effect images, or Pencil page files in this global stage.
23.1. Prefer `@product-design` `ideate` for shared style comparison, theme-route narrowing, and hierarchy-system selection. Do not use it to produce PRD prose, implementation RDs, page design drafts, module freeze decisions, or Flutter code.
23.2. Do not accept a generated representative sketch as global-stage evidence unless a legacy branch already produced it, its automatic Product Design QA receipt exists, and its `commercial_surface_gate.result=passed` receipt exists. If the gate fails, record `revision_target=global_style_scheme` and revise the selected style scheme instead of confirming a page-level product design direction.
24. If a `global_style_scheme` recommendation artifact exists but the user has not yet confirmed the final style direction, set or keep `current_stage=design_recommendation_ready`, record `required_inputs=product_design_final_direction_confirmation`, and stop. In manual mode, this confirmation step should expose exactly three recommended theme and style schemes with one primary recommendation instead of a hidden answer or page comp. In `--full-auto`, if one clear primary recommendation exists and no competing approved direction remains unresolved, auto-confirm that style direction and continue.
25. If the workflow needs richer campaign, hero, mood-board, or commercial visual evidence before the representative sketch direction can be chosen confidently, keep the run on the shared-direction branch, read `creative-production-branch.md`, route into the pre-direction Creative Production branch, and return to Product Design confirmation with that evidence.
26. If the user already supplied a concrete approved visual source, use that source only as evidence for the selected `global_style_scheme` after Product Design brief playback. Do not call `gpt-image-2-generator` for a global representative final effect image; record `current_stage=product_direction_confirmed` or queue that stage according to the confirmation gate once the style scheme itself is selected.
27. If the final global style direction is confirmed after the style-scheme recommendation pass, do not route shared/global scope to `effect-image-to-ui-sheet-atlas`. Queue `product_direction_confirmed` and proceed to `DESIGN.md` so the global freeze captures theme, hierarchy, public shell, component family, minimal-copy rules, and interaction principles only. Module-scoped `effect-image-to-ui-sheet-atlas` remains available later after an active module `impl.md` and native module HTML prototype are accepted.
27.1. If the current scope is at `ui_sheet_atlas_analysis_ready`, do not advance to `DESIGN.md` yet. First require atlas-analysis confirmation in manual mode, then rerun `effect-image-to-ui-sheet-atlas` to generate the atlas bundle.
27.2. If the current scope is at `ui_sheet_atlas_ready`, do not advance to `DESIGN.md` yet. First require solid-atlas-bundle confirmation in manual mode, then route to `$imagegen` background removal using the chosen preset background color.
27.3. After `$imagegen` produces the transparent atlas result, queue `pending_next_stage=ui_sheet_atlas_transparent_ready`, record the transparent atlas path plus validation result, and stop for confirmation in manual mode.
27.4. If the current scope is at `ui_sheet_atlas_transparent_ready`, do not advance to `DESIGN.md` yet. First require transparent-atlas confirmation in manual mode, then route to `ui-sheet-atlas-slicer` in deterministic minimum-cell mode.
27.5. When `ui-sheet-atlas-slicer` succeeds for the shared representative page, queue `pending_next_stage=product_direction_confirmed` and record the slice manifest before continuing.
27.6. Do not accept a final effect image, implementation-stage module effect image, atlas-analysis input, or atlas-preparation input unless `commercial_surface_gate.result=passed` has already been recorded for the matching image. If the image fails the gate, route back to same-stage image revision and keep atlas work blocked.
28. After `product_direction_confirmed`, read both `design-quality-guidance.md` and `design-md-template.md`, then route to output a root-level `DESIGN.md` before entering any later structured design-source normalization, shared freeze, module design freeze, architecture, or code.
28.1. Unless the user explicitly requested another language, `DESIGN.md` and its human-readable sections must be written in Simplified Chinese.
29. When `DESIGN.md` is successfully written at the project root, queue or promote `current_stage=design_md_ready`.
29.1. Do not route global/shared scope to `effect-image-to-pencil-design`. Page-level Pencil design source is mandatory only for module scope after the module HTML, effect-image, atlas, and display-evidence gates are accepted.
29.1.1. Do not accept a Pencil design source as global-stage evidence or as a shared-freeze prerequisite. If a Pencil file exists from a legacy branch, it may only inform the selected `global_style_scheme` after provenance and gate receipts are recorded.
29.2. Do not route shared/global scope through shared runtime-asset normalization, shared page bitmap generation, or page atlas slicing. Reusable asset decisions that affect the theme may be recorded as style guidance, but actual page bitmap assets belong to module scope.
29.3. If a legacy shared Pencil design source, shared effect image, or shared atlas already exists, do not use it as a required global freeze input. It may be referenced only as non-authoritative evidence for the selected theme/style scheme after its provenance and gate receipts are recorded.
30. Do not enter shared freeze, module design freeze, architecture, or code before the global visual design direction has been brainstormed, the common public shell has been agreed, the final style direction has been confirmed, `global_style_scheme.status=selected`, and `DESIGN.md` exists.
31. Before shared freeze preparation, verify that `DESIGN.md` explicitly covers at least task hierarchy, first-screen CTA posture, interaction and feedback rules, responsive strategy, critical states and edge cases, content tone when those concerns are relevant to the product, the frozen base design viewport or device preset, platform-aware information-density posture for major surfaces, and the selected `global_style_scheme`. If that coverage is weak, route back to revise `DESIGN.md` before continuing.
31.1. Do not require shared-scope bitmap assets after global style selection. If the project needs non-standard bitmap assets, defer the decision to module scope after module HTML, module effect-image, atlas, and Pencil review evidence exist.
31.2. Before shared freeze preparation, run one explicit copy-minimization check on the shared shell principles, theme copy posture, and representative content rules. If first-screen understanding still depends on repeated explanation, multi-paragraph helper copy, or redundant descriptive text, route back to revise the style scheme and `DESIGN.md` before continuing.
31.2.1. Do not promote shared freeze unless `minimal_default_copy_contract.status=passed`. That contract must record the `explanatory_copy_budget`; if the result is `explanation_overload`, route back to reduce visible explanation or move explanation behind disclosure before continuing.
32. If the request is primarily about campaign visuals, ads, mood boards, launch assets, social cards, hero directions, scenes, shots, logos, or other asset-oriented creative work, read `creative-production-branch.md` and route into `Creative Production:explore` once the brief is specific enough.
32.1. If the workflow needs a Product Design-owned coded review prototype from a selected visual target, route to `@product-design` `prototype` or `image-to-code` only after the design brief is confirmed and the visual target is explicit. Use `prototype` when the source form is still mixed or tool choice is unresolved; use `image-to-code` when the selected visual target is already an image, mockup, screenshot, or equivalent screen reference.
32.2. Do not use `@product-design` `prototype` or `image-to-code` as a replacement for the mandatory module native `HTML/CSS/JS` prototype gate, for Pencil design generation, or for Flutter implementation. Those remain separate workflow responsibilities.
33. Before final product direction confirmation, treat the Creative Production branch only as direction input. After `design_md_ready`, treat it as the post-direction asset-production branch.
34. If the user already selected an asset family or asks directly for one of the focused Creative Production paths, skip the generic chooser and route directly to the scope-matched explorer: `moodboard-explorer`, `ads-explorer`, `offer-explorer`, `scene-explorer`, `shot-explorer`, or `logo-explorer`.
35. If the user needs publish-bound marketing finish, exact-size social outputs, or premium visual polish while preserving exact text, logos, charts, data, safe zones, and dimensions, route to `Creative Production:generative-polish` only after one direction or deterministic base is already selected.
36. Do not use the Creative Production branch to replace Product Design brief confirmation, product-surface direction selection, `DESIGN.md`, or shared freeze. Treat it as an asset branch, not as a shortcut around the app-design workflow.
37. Do not route broad marketing or campaign exploration straight to `gpt-image-2-generator` when the Creative Production branch is available and the user needs reviewable creative routes, boards, or publish-safe asset production. Use representative-sketch generation first for the app-preview branch, and use final effect-image generation only after the sketch is confirmed or when the user explicitly wants lightweight page evidence instead of the richer asset workflow.
38. If a Creative Production run surfaces a materially different visual direction from the already confirmed product direction, stop downstream freeze or implementation promotion, route back to Product Design confirmation, and revise `DESIGN.md` before continuing.
39. Final effect-image generation is mandatory only for module/page design scope. The shared/global route must not enter representative sketch generation, page effect-image generation, atlas preparation, or Pencil page design; it must stop at the confirmed `global_style_scheme`, optional `global_style_experience_image` through `Product Design:ideate` with `non_page_design_evidence=true`, plus `DESIGN.md`.
40. In manual mode, reuse the approved Product Design visual target and reference images only as style evidence for the global scheme. If no approved style evidence exists yet, stop and route back to the Product Design recommendation branch instead of inventing a new visual baseline.
41. Before module representative final effect-image generation, verify that the selected final-generation path is available and authenticated. If access, credentials, or upstream visual generation capability is missing, stop and record the branch-specific blocker. Then route the module representative sketch to `Product Design:ideate`, and route the module final effect-image direction pass to `Product Design:ideate` before freezing the chosen baseline. The workflow must select or revise a Product Design-generated candidate instead of sending the module final effect image to `gpt-image-2-generator`.
42. Before any module-stage representative-sketch or final effect-image generation request is sent, read the frozen global style packet and extract the applicable visual constraints for the target page or module. Generated sketches and final effect images must explicitly inherit the confirmed product design direction, the frozen base design viewport, `DESIGN.md`, PRD constraints, public-shell decisions, art direction, visual system, CTA posture, palette direction, typography mood, component family cues, image-treatment posture, target page purpose, and any first-screen task priority that materially affects the page when those values exist. The module stage must remain within `@product-design` design control and must read as mature commercial product UI/UX instead of concept art or marketing collateral. Keep the frozen design width unchanged, treat the frozen viewport height as the minimum image height, and allow taller outputs only when the target page or region is intentionally scrollable. The atlas-preparation node must first produce a matching atlas extraction analysis whose runtime-data regions remain excluded and whose exportable visuals come only from the post-analysis `atlas_required` set. Only after that analysis is confirmed may it generate the solid-background atlas bundle. That atlas bundle must use rectangular non-overlapping cells, carry restore bounds for design restoration, and record one chosen least-conflicting background color from the preset atlas background palette. After the solid atlas is confirmed, route to `$imagegen` to remove the background and confirm that transparent result before deterministic minimum-cell slicing. If generation fails and a retry is needed, resend the full original prompt plus the same frozen constraints and any blocker-specific repair notes; do not simplify, shorten, or strip prompt content for the retry.
42.1. Every such module request must also include an explicit negative constraint set: not marketing visual, not ad creative, not poster, not landing-page hero, not concept board, not wireframe-like draft, and not a requirements-explanation mock. The intended output is always an in-product commercial app screen unless the route has already switched into the dedicated Creative Production branch.
42.2. For shared/global scope, do not send page-generation requests. For module scope, do not send the request unless the workflow record already contains an accepted native HTML module prototype receipt for the active module.
43. In manual mode, save module representative sketches into the module effect-image directory under `docs/project/` using the corresponding page name plus a `-draft` suffix, require that sketch to come from `Product Design:ideate`, run one automatic `@product-design` QA pass on that generated sketch, index both the sketch path and QA result in workflow state, write the generated image row into `docs/project/assets/global-asset-catalog.json`, set the workflow to wait for user confirmation or revision feedback, and do not generate the representative final effect image or the remaining required page effect images yet.
43.1. Any human-readable QA summary, review note, or confirmation text generated in this step must use Simplified Chinese unless the user explicitly requested another language.
44. In manual mode, after the module representative sketch is confirmed, route the module final effect-image direction pass to `Product Design:ideate`, select or revise a Product Design-generated candidate, and freeze the selected module final effect image as the module visual baseline only after one automatic `@product-design` QA pass. Write the generated image row into `docs/project/assets/global-asset-catalog.json`, generate the matching atlas extraction analysis, stop for analysis confirmation when required, then generate the matching solid-background UI-only atlas plus atlas manifest plus atlas slicing config, confirm that solid atlas bundle, keep atlas background removal plus slicing on the deterministic asset path, route to `$imagegen` for background transparency removal, confirm the matching UI-only transparent atlas result, and only then continue into deterministic minimum-cell slicing. Delete previously generated alternative sketches or effect images that are no longer part of the frozen packet, update indexes plus the same catalog rows to keep only the active baseline, and record that alternative-direction browsing is closed. Do not reopen Product Design direction browsing or generate other-direction images inside the same design cycle.
45. In `--auto` or `--full-auto`, once the module representative effect-image direction is chosen, run one automatic `@product-design` QA pass on every generated image before freezing it as the module visual baseline, write the generated image rows plus QA results into `docs/project/assets/global-asset-catalog.json` or the workflow record, and continue without inventing an extra all-page effect-image stage.
46. All newly generated module-stage effect images must be light-mode visuals. Do not generate dark-mode freeze evidence unless the user explicitly requests a dark-only or dual-mode design cycle.
47. Before any global design freeze or module design freeze, keep `@product-design` as the primary design controller. The global freeze normalizes the confirmed `global_style_scheme` and `DESIGN.md`; module freeze normalizes approved module images, atlas evidence, Pencil design, and any still-required bitmap generation.
48. Before module Pencil design execution or later bitmap generation, require a root-level `DESIGN.md`, the frozen global style packet, and the active module design package.
51. If the confirmed `DESIGN.md` and selected `global_style_scheme` must become reusable fixed global guidance, use `design-preview-to-global-guidelines` only for theme/style normalization, then record `global_guidelines_frozen` as `pending_next_stage` until the user confirms.
52. Before `design_freeze_ready` and the shared/global design freeze complete, do not advance into any module-related workflow. Module `impl.md` generation, module implementation-stage effect-image generation, module freeze, module architecture, and module code all stay behind the shared freeze gate.
53. If shared design has a candidate direction, confirmed `DESIGN.md`, and a selected `global_style_scheme` but no explicit freeze decision, use `flutter-design-freeze-gate` directly. Before freezing, explicitly verify that the selected scheme still preserves the approved style direction, theme system, public shell contract, shared public component families, task hierarchy, interaction feedback behavior, responsive strategy, and critical-state language.
54. If `flutter-design-freeze-gate` finds hierarchy, task guidance, typography, contrast, CTA clarity, state coverage, feedback consistency, responsive strategy, or content-tone coverage still weak, do not route to freeze. Allow one shared revision pass through `global_style_scheme` and `DESIGN.md` refinement, then stop unless the user explicitly restarts the design cycle.
55. Once the shared/global design freeze is complete and the shared bootstrap-critical baseline is explicit enough, prefer `flutter-init` immediately instead of waiting for later module architecture milestones.
56. If the shared bootstrap-critical baseline is ready and the target project has not been scaffolded yet or the workspace does not contain sibling `skills/flutter-dev/`, use `flutter-init` before feature-module implementation begins.
57. `flutter-init` must stop at initialization boundaries. It may create the directory skeleton and sibling `skills/flutter-dev/` container, but it must not implement bootstrap code, shared wiring, feature code, page code, or module-specific business logic. Unless the user explicitly overrides the environment contract, route initialization and later command guidance under the assumption that Flutter is managed through `fvm`.
58. If `flutter-init` has completed and the directory skeleton plus sibling `skills/flutter-dev/` exist, record `project_initialized` as `pending_next_stage`.
59. After `project_initialized` is confirmed, route to the separate bootstrap code stage.
60. `project_initialized` is directory-only. Do not treat it as proof that bootstrap code, shared wiring, or any feature implementation already exists.
61. If bootstrap code has landed and the required global public baseline is actually present on disk, record or queue `bootstrap_code_ready`.
62. If the shared/global design freeze is complete but detailed modules and executable module `impl.md` paths do not exist yet, first verify that the confirmed Product Design clarification packet for module splitting is present. If it is missing, route back to `@product-design` scope clarification and stop. Otherwise use `flutter-rd-module-splitter` to create module index rows and implementation-final `impl.md` documents in one pass. Those module documents must explicitly inherit the frozen shared design direction, shared interaction principles, shared public shell decisions, shared component constraints, platform-specific interaction expectations, and approved information-density posture.
62.1 Before accepting a module split, verify that each page family has exactly one owning module.
62.2 If two candidate modules still describe the same primary user job, same core data owner, and same state lifecycle, do not approve the split yet. Keep the pages in one module unless a real permission boundary, release slice boundary, or independent backend ownership proves the separation.
63. If multiple feature modules depend on one shared route host, root redirect layer, tab shell, or shell-level state, allow `flutter-rd-module-splitter` to split an `app-shell` or `root-shell` module first instead of burying that responsibility inside feature modules, but still only after the shared/global design freeze is complete.
64. If a selected module lacks an executable `docs/project/modules/<module>/<module>.impl.md`, route back to `flutter-rd-module-splitter` for a scope-matched regeneration instead of opening a separate refinement stage.
65. Module implementation preparation consumes the executable `impl.md` produced by `flutter-rd-module-splitter`. Treat that `impl.md` as a detailed module task implementation document constrained by the frozen shared design and interaction principles. Do not require or create `docs/project/modules/<module>/<module>.ui-ux.md`.
66. When `flutter-rd-module-splitter` produces executable module `impl.md` documents, keep `current_stage` at the last confirmed stage, queue `pending_next_stage=module_impl_docs_ready`, queue each generated module's `impl_status=implementation_final`, and wait for confirmation before the active module's native HTML preparation. In `--auto` or `--full-auto` mode, auto-apply those queued updates, select the first unfinished module in the confirmed serial module order, and continue directly into that active module's native HTML path.
