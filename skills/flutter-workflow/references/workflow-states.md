# Workflow States

Use this reference when classifying `current_stage`, module maturity, or allowed next moves.

Use one state per module:

| State | Meaning | Allowed Next Move |
| --- | --- | --- |
| `idea_sketch_brainstorming` | The input is still a rough idea, reference bundle, or abstract product direction that is not concrete enough to reason about through pages yet. The orchestrator must brainstorm the core product path, generate `1-3` key-page sketches, confirm product direction plus page structure plus broad visual style, and write `docs/project/idea-sketch-brief.md` before PRD work may begin. | sketch confirmation, brief artifact write, then `requirements_brainstorming` |
| `requirements_brainstorming` | The workflow has a confirmed `docs/project/idea-sketch-brief.md` or an already-concrete feature request, but no PRD artifact is ready yet. The orchestrator must complete the requirement-space brainstorming needed for PRD quality: goals, users, scope, rules, risks, assumptions, and unresolved questions. | internal requirements-to-PRD flow, then `flutter-prd-rd-writer` when ambiguities are resolved |
| `prd_ready` | PRD or feature brief exists, the question ledger is resolved strongly enough, and the PRD completeness gate has passed | `flutter-prd-rd-writer` for global technical baseline generation |
| `technical_baseline_ready` | Global architecture, package stack, backend collaboration, and delivery assumptions exist, but the target design-device preset plus design viewport and `image_output_scale`, the Product Design brief, and the frozen shared visual direction are not yet ready. | freeze design-device preset, design viewport, and `image_output_scale`, then `@product-design` `get-context` |
| `shared_design_direction` | The target design-device preset, design viewport, and `image_output_scale` are frozen for this design cycle, `@product-design` has confirmed the design brief or is actively resolving it from the PRD and technical baseline, and the workflow is ready to select one Mobbin-backed `global_style_scheme` rather than produce shared/global page design drafts. The global style scheme must record `theme_and_style_only=true`, `no_global_page_design_draft=true`, and `page_design_deferred_to_module_stage=true`. When the direction needs a more intuitive review surface, this stage may also prepare one `global_style_experience_image` with `non_page_design_evidence=true` as a non-page style board. When richer commercial visual evidence is useful, this stage may also open the pre-direction Creative Production branch before the final style direction is locked. | public-shell confirmation, Mobbin-backed style recommendation, then `design_recommendation_ready` |
| `shared_html_prototype_ready` | Legacy-only shared-page state. The current global style-only workflow must not require or create a shared native `HTML/CSS/JS` page prototype before global freeze. If this state appears from an older run, treat its artifact as non-authoritative style evidence and keep `no_global_page_design_draft=true`. | migrate evidence into `global_style_scheme`, then `shared_design_direction` or `design_recommendation_ready` |
| `design_recommendation_ready` | `@product-design` has produced one or more Mobbin-backed theme and style directions, including one primary recommendation when the run is in manual mode, and the workflow is waiting for human confirmation or revision selection before locking `global_style_scheme.status=selected`. A `global_style_experience_image` may be shown here only as non-page visual style evidence with `non_page_design_evidence=true`. No page sketch, final effect image, atlas, or Pencil page file is required at this global stage. In `--auto`, the orchestrator may adopt the primary style recommendation directly when no blocker remains. In `--full-auto`, that same primary recommendation may also satisfy the final-direction confirmation gate when no competing approved direction remains unresolved. | human confirmation or recommendation revision, then `product_direction_confirmed` |
| `ui_sheet_atlas_analysis_ready` | The current shared or module scope has produced a reviewable atlas extraction analysis from the confirmed effect image. That analysis already classifies `flutter_native`, `atlas_required`, and `placeholder_only` regions, excludes dynamic-data-driven visuals, and defines the rectangular non-overlapping atlas cell plan that downstream generation must follow exactly. This analysis must be confirmed before atlas image generation begins. | human confirmation, then `ui_sheet_atlas_ready` |
| `ui_sheet_atlas_ready` | The current shared or module scope has produced a reviewable solid-background atlas bundle containing the target effect image, one UI-only atlas image, the chosen background-key color, one atlas manifest, and one atlas slicing config, all grounded in the confirmed atlas extraction analysis. This bundle must be confirmed before the workflow enters the background-removal step. | human confirmation, then `ui_sheet_atlas_transparent_ready` |
| `ui_sheet_atlas_transparent_ready` | The confirmed solid-background atlas has already passed through `$imagegen` background removal and now has one reviewable transparent atlas result ready for slicing. This stage records the matching UI-only transparent atlas artifacts. This transparentized atlas must be confirmed before deterministic minimum-cell slicing begins. | human confirmation, then `ui_sheet_atlas_slices_ready` |
| `ui_sheet_atlas_slices_ready` | The confirmed transparent atlas bundle has already been cut by the dedicated slicing node in minimum-cell mode, and the workflow now has slice files plus a slice result manifest ready for downstream Pencil restoration or later design-source work. For module pages, overlay UI slices are valid only when the same page's `page_base` slice contract already exists. | `display_evidence_pack_ready` or `pencil_design_source_ready` depending on module scope |
| `product_direction_confirmed` | The user or deterministic auto gate has confirmed `global_style_scheme.status=selected` as the final product style direction. The record may include one `global_style_experience_image`, but it must keep `theme_and_style_only=true`, `no_global_page_design_draft=true`, `non_page_design_evidence=true`, and `page_design_deferred_to_module_stage=true`; this confirmation is not page design evidence and does not require representative sketches, final effect images, atlas bundles, or Pencil page files. | output `DESIGN.md` |
| `design_md_ready` | The confirmed global style direction has been written into `DESIGN.md`-ready downstream inputs: theme, hierarchy, public shell, component family, minimal-copy, information-density, and interaction principles. Asset-oriented requests may also open the post-direction Creative Production asset-production branch from here without weakening the product-design freeze rules. | optional post-direction Creative Production branch, then `global_guidelines_frozen` or `flutter-design-freeze-gate` |
| `global_effect_image_pilot_ready` | Legacy-only global image state. Global effect-image states are legacy-only and must not be used as required global freeze gates in the current style-only workflow. If an artifact exists from an older run, record it as non-authoritative visual evidence, not page design evidence. | migrate evidence into `global_style_scheme` or module scope |
| `global_effect_images_ready` | Legacy-only global image state. Shared light-mode effect images are not required for global freeze and must not replace `global_style_scheme` plus `DESIGN.md`. Module-scoped effect images may still reuse this state name only after the active module has entered the implementation-stage visual restoration loop. | `display_evidence_pack_ready`, `pencil_design_source_ready`, or module-scoped freeze work |
| `pencil_design_source_ready` | The active module after it has already entered implementation has routed through `effect-image-to-pencil-design`, built the mandatory high-fidelity Pencil design source, consumed the confirmed atlas slice outputs for atlas-backed visuals, compared it against the approved effect image, completed one confirmed repair pass, and passed human acceptance for the Pencil file. This review result is now the basis for deciding whether any new bitmap generation is still required. Shared/global Pencil page files are not required by the global style-only workflow. | `module_asset_resources_ready`, `global_guidelines_frozen`, `module_design_frozen`, or `flutter-design-freeze-gate` depending on scope |
| `shared_asset_resources_ready` | Legacy-only shared-page resource state. Shared page bitmap generation is not required for global freeze in the current style-only workflow. Reusable asset decisions may be noted in the global style scheme, while concrete page bitmap assets belong to module scope. | migrate evidence into `global_guidelines_frozen` or module scope |
| `global_guidelines_frozen` | The confirmed `DESIGN.md` and selected `global_style_scheme` have been converted into frozen shared guidance and explicit global public component decisions. No module page design is included at this stage. This is the first internal half of the shared freeze gate rather than a separate user-facing destination. | `flutter-design-freeze-gate` |
| `design_freeze_ready` | The shared theme/public-shell design packet, `DESIGN.md`, `global_style_scheme`, hierarchy contract, minimal-copy contract, and interaction-principle contract are ready for approval before any module-related workflow may begin. The shared/global freeze covers theme and public shell only, not page design. Once this shared freeze gate passes, the workflow should prefer early project initialization plus bootstrap preparation and may also begin module implementation-document generation. | `flutter-init`, bootstrap preparation, or `flutter-rd-module-splitter` |
| `module_impl_docs_ready` | The shared design freeze is already complete, and the active module's detailed task implementation document `impl.md` has been generated under the frozen shared design and interaction principles. That `impl.md` fixes the module function, key states, task path, and non-display behavior boundaries strongly enough to drive the module native-HTML confirmation stage and the later implementation-stage visual restoration loop. | `module_html_prototype_ready` |
| `module_html_prototype_ready` | The active module has produced a reviewable native `HTML/CSS/JS` static prototype that shows the module pages' layout, hierarchy, and visual grouping under the frozen shared design constraints. This stage is the mandatory module confirmation gate before atlas analysis, atlas preparation, atlas background removal, atlas slicing, and module Pencil design work. | human confirmation, then `ui_sheet_atlas_analysis_ready` |
| `module_asset_resources_ready` | This is an implementation-stage milestone only. After the active module has already entered the visual restoration loop and the module Pencil design review was accepted, the workflow processed each target module page one by one: it identified that page's remaining new bitmap assets, resolved or queued reuse questions through the global asset registry, waited for explicit confirmation of that page's post-review bitmap list, generated each approved resource independently through `$imagegen`, and saved the final approved asset files that the module's structured design source and later code must reference directly. | `pencil_restoration_ready` or continue `implementing` depending on scope |
| `display_evidence_pack_ready` | The active scope now has enough readable visual evidence for fidelity-critical regions: main preview, required detail views, key state views, scroll evidence, overlay evidence, confirmed atlas-analysis evidence, confirmed transparent-atlas evidence, and confirmed atlas-slice evidence where needed. The workflow has confirmed that later freeze, architecture, and display restoration do not need to guess missing structure or state language. | `flutter-design-freeze-gate` or `pencil_design_source_ready` depending on scope |
| `module_design_frozen` | The active module's implementation-final `impl.md`, confirmed native HTML prototype, implementation-stage effect-image evidence, selected Pencil-backed structured design-source packet, explicit module scope decisions, and high-fidelity restoration target are frozen strongly enough to enter implementation. | `flutter-design-source-control`, `impl_rd_ready`, or `pencil_restoration_ready` |
| `impl_rd_ready` | The active module's implementation RD is implementation-final or landed, references the frozen structured design-source packet and global technical baseline, and is confirmed as implementable. | `flutter-uiux-to-architecture` |
| `pencil_restoration_ready` | The active module has already routed through `pencil-design-to-flutter-restoration` and now has a restoration contract that explicitly classifies Flutter standard-library fit, third-party plugin fit, and approved bitmap-asset usage for the frozen Pencil design source. | `flutter-uiux-to-architecture` |
| `architecture_ready` | Tokens, assets, components, screen plan, non-native visual asset strategy, and scaffold contract exist, but the workflow may still need to confirm that the page-level display restoration contract is explicit enough for implementation. | `display_restoration_blueprint_ready` |
| `display_restoration_blueprint_ready` | The active module now has a concrete display-layer decision table and a page-level `display_restoration_blueprint` covering region ownership, scroll and sticky rules, state slots, asset bindings, and fidelity locks. This is the implementation-facing display contract and is mandatory before code execution begins. | implementation entry once `bootstrap_code_ready` and other implementation prerequisites already exist |
| `project_initialized` | `flutter-init` has only created the directory skeleton and sibling `skills/flutter-dev/` container, but has not started bootstrap code, shared wiring, or any feature implementation code. Unless explicitly overridden, the workflow should already treat `fvm` as the default Flutter environment contract from this point forward. | bootstrap code generation |
| `bootstrap_code_ready` | Global bootstrap code exists: app entry, root app shell, shared wiring, and core startup scaffolding are landed, but feature-module implementation has not started yet. Flutter and Dart execution guidance should still default to `fvm` unless the user explicitly changed the environment contract. | `@superpowers` `Spec` for implementation entry |
| `implementing` | Implementation entry begins only after bootstrap code is ready, `display_restoration_blueprint_ready` is already confirmed, and the module `impl.md` is executable. The module must first pass through `@superpowers` `Spec`, then `@superpowers` `Plan`, and only then execute code work. The module-level implementation loop is fixed and serial: confirm `impl.md` as the functional contract, confirm the native `HTML/CSS/JS` static prototype as the layout evidence baseline, confirm the atlas extraction analysis bundle, the solid atlas-preparation bundle, the transparent atlas result, and the atlas slices as the page-image evidence baseline, execute module Pencil review plus one focused repair pass when needed, generate only the still-required module bitmap assets, honor the frozen restoration contract plus `display_restoration_blueprint`, land non-display-layer code against that blueprint, and restore the display layer from that same blueprint last. The orchestrator processes modules one by one rather than opening parallel module lanes; display-layer work must inspect the corresponding native HTML prototype, implementation-stage module effect image, atlas manifest, slice manifest, frozen Pencil design, `display_restoration_blueprint`, and frozen design evidence first when such evidence exists. When implementation screenshots exist, the workflow should route through render-vs-frozen-design QA before final helper review and human acceptance. | `@superpowers` `Spec` -> `@superpowers` `Plan` -> serial module implementation execution -> `flutter_render_vs_frozen_design_qa` -> optional blocking `@product-design` `design-qa` -> final human visual inspection |
| `flutter_render_vs_frozen_design_qa` | The active module's rendered Flutter result has been compared against the frozen design source, approved effect images, approved generated asset files, and `display_restoration_blueprint`, and the workflow has an explicit render-level fidelity QA result before final human acceptance. | optional blocking `@product-design` `design-qa`, then final human visual inspection |
| `module_done` | Shared freeze is confirmed, module implementation docs are landed, the selected structured design source is frozen, `display_restoration_blueprint_ready` is confirmed, code is landed, render-vs-frozen-design QA has already run, and the user has accepted the final human visual inspection result. human visual inspection remains the final acceptance owner, and final app-page acceptance must happen only at the end of the workflow. | maintain index only |
