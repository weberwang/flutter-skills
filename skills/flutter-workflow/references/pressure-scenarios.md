# Pressure Scenarios

Use this reference when the user request resembles one of these examples or tries to skip a workflow gate.

- User gives a raw one-line requirement with no PRD: enter `requirements_brainstorming`, build the question ledger, confirm `page-navigation-flow`, and generate the PRD before technical baseline work.
- User says "skip PRD and decide the technical stack first": block. Raw demand must pass requirements brainstorming and PRD generation before technical baseline, shared design freeze, or module document generation.
- User says "implement this home page directly" with only a PRD: route to `flutter-prd-rd-writer`, not module document generation or code.
- User says "split modules first, choose packages later": block and require baseline architecture and package decisions first.
- User says "split modules now and fill in journeys or critical states later": block. Require the confirmed Product Design clarification packet first so module splitting does not infer journeys, page families, or critical states from page count alone.
- User says "design globally first and confirm the viewport later": block. Freeze the target design-device preset, design viewport, and `image_output_scale` before global design exploration, effect-image generation, or shared freeze preparation.
- User says "skip Mobbin and jump straight to direction": block. The workflow must complete a Mobbin-first inspiration pass before direction recommendation or final direction confirmation.
- User says "confirm direction directly without Product Design brief playback": block. Final product design direction confirmation requires a prior `@product-design` brief-confirmation step and an approved direction input.
- User says "freeze the design draft now": route to `flutter-design-freeze-gate`.
- User says "the shared effect image looks close enough, just freeze it": block and route back through exactly one shared revision pass.
- User says "make the happy path pretty first and handle empty/error/loading later": block when those states are already relevant to the confirmed scope. Required states belong to the active design target before freeze; only additional unconfirmed states may be appended later after explicit human confirmation.
- User says "put all the content on the page first and compress it later": block. Require a platform-specific density decision first, including what must stay visible immediately and what must be deferred, collapsed, paged, or moved to another surface.
- User says "show less content now and backfill later if it feels empty": block. Density must follow the target platform, task priority, and state needs, not a temporary placeholder posture.
- User says "generate the effect image first, we can decide the common shell later": block. Module effect-image generation still requires explicit agreement on the common public shell first.
- User says "skip final product direction confirmation and generate images directly": block. Effect-image generation still requires explicit final direction confirmation after the Product Design direction step.
- User says "just assume the default phone size" in manual mode: do not silently assume. Offer the common iPhone presets first unless the user already approved a viewport.
- User says "skip DESIGN.md after direction is confirmed": block. Final product direction confirmation must be followed by `DESIGN.md` before shared freeze or module-stage design-source work.
- User says "show me the recommendations first and I'll choose": allow. Route to the `@product-design` recommendation branch, stop at `design_recommendation_ready`, and require human confirmation before promoting `product_direction_confirmed`.
- User says "generate all page effect images now" before the first module final effect image is confirmed: block. Confirm the first module final effect image first, then expand to the remaining required module pages when the route still requires them.
- User says "retry image generation with a much shorter prompt": block that shortcut. Retries must preserve the full original prompt and frozen constraints; only blocker-specific repair notes may be added.
- User says "this direction is already confirmed, but let's browse other directions too": block. Once the representative or approved direction effect image is confirmed, the current design cycle must keep that visual baseline frozen unless the user explicitly rejects it or restarts the cycle.
- User says "skip global effect images and freeze directly": allow on the shared/global route. Shared freeze is style-only and does not require page effect images; module effect-image work happens later in module scope.
- User says "tone can be decided later, just get the UI out first": block when brand tone or UX writing tone materially affects CTA confidence, onboarding, empty states, or recovery states.
- User says "include page design in the shared freeze": block. Shared/global freeze is limited to theme, public shell, shared component families, and shared interaction principles; page design belongs to the later active-module path.
- User says "use the tool's own internal taste guide to fill missing design details": block. Design guidance may come only from the frozen global design, the active module design package, frozen design-source artifacts, and `DESIGN.md`.
- User says "write impl.md first and freeze global design later": block. All module-related workflow must stay behind the shared/global design freeze.
- User says "ignore platform and density details during module splitting; we can tune that during implementation": block. Module docs must already encode platform-aware navigation, feedback, and information-density constraints before freeze and implementation.
- User says "implement multiple modules in parallel": block in the default workflow. Modules must follow the confirmed serial module order after the shared/global design freeze.
- User says "modules are split, now refine only the home module in isolation": block the separate refinement stage. Route back to the combined executable module document generation step for a scope-matched regeneration of `home.impl.md`.
- User says "continue auto after module docs" but the module index, executable module `impl.md`, frozen structured design-source packet, or `display_restoration_blueprint_ready` gate is missing: treat that as a real blocker, record `not_executed`, and stop instead of reconstructing fake progress.
- User says "build the display layer first and fill in behavior later": block that path for ordinary page modules. Keep the module loop as `module scope -> final effect image -> impl -> native HTML prototype -> design source -> module freeze -> restoration and blueprint -> non-display code -> display restoration`.
- User asks why a root navigation host was split separately: explain that an `app-shell` module is valid when shared route hosting, root redirects, or shell-level state has independent implementation value.
- User says "the docs are final, start coding": require the queued freeze and implementation-readiness promotions to be confirmed first.
- User says "we can make it high fidelity during implementation": allow only the implementation-stage visual restoration work that is already defined by the frozen module contract. Do not let implementation silently replace missing module states, missing shared constraints, or a vague module freeze packet.
- User says "Phase 1 can skip atlas evaluation because launch uses native Flutter first": block. Launch still must complete `launch_asset_enhancement_resolution`; only the final result may be `no_atlas_required`.
- User says "Phase 1 already has a Pencil design source, so atlas analysis can be skipped": block when any region still needs bitmap fidelity. Once launch asset-enhancement resolution marks a region `atlas_required`, atlas analysis, generation, background processing, slicing, and asset integration become mandatory for that region.
- User says "run `flutter-workflow --auto`": keep advancing through technical baseline, `@product-design` brief confirmation, `DESIGN.md`, shared freeze, project initialization, bootstrap code generation, and then the serial per-module loop of `final effect image -> impl.md -> native HTML prototype -> selected design-source branch execution -> module freeze -> branch-matched restoration -> asset-enhancement resolution -> architecture/blueprint -> @superpowers Spec -> @superpowers Plan -> code implementation -> flutter_render_vs_frozen_design_qa`. When any region is marked `atlas_required`, do not skip atlas analysis, generation, background processing, slicing, or asset integration for that region.
- User says "auto mode can choose the default viewport for me": allow. If no frozen design viewport exists yet, use `390 x 844 px`.
- User says "generate real-device effect images during module refinement too": do not do that by default; require an explicit evidence need from the current route.
- User says "the primary platform is Android": record that as explicit `platform_identifier`, and validate on Android only instead of expanding the workflow into optional Windows or Web verification.
- User says "Android has multiple devices": stop and wait for explicit device selection instead of auto-picking one.
- User says "Android has no device available": start an emulator and use it as the primary-platform validation target.
- User says "the project is initialized, start feature coding": first verify that the separate bootstrap code stage has landed. Directory creation alone does not authorize feature-module implementation.
- User says "adjust button hierarchy during implementation": route to `flutter-design-source-control`.
- User says "implement the screen layer now": strictly route to `@superpowers` and require the corresponding page image plus frozen design evidence to be inspected first when that evidence exists.
- User says "start implementing this feature module now": first verify that the required global public code baseline is already landed; if not, block module implementation and land the missing global code first.
- User says "start coding directly" before `@superpowers` gates exist: block direct execution and require `@superpowers` `Spec` first, then `@superpowers` `Plan`, before any code implementation begins.
- User says "auto mode should not stop at every ordinary review gate": allow. `--auto` should auto-confirm validated orchestrator-owned transitions; `--full-auto` may additionally consume deterministic human-facing gates when the control contracts prove there is exactly one supported default.
- User says "this module is done, what next" while `--auto` or `--full-auto` is active and other target modules remain: do not stop to ask. Select the next module in the confirmed serial module order, update the workflow record, and continue automatically.
- User says "the code is done, check whether it matches the design": first run `@product-design` `design-qa` when a source visual target and implementation screenshots both exist, and run the `flutter_render_vs_frozen_design_qa` stage through `flutter-design-parity-reviewer` before final human visual inspection.
