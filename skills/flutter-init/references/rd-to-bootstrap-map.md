# RD To Bootstrap Map

## 用途

- 把 `flutter-prd-rd-writer` 产出的 RD 章节映射成 Flutter 初始化动作
- 在真正开始 `flutter create`、目录搭建、依赖接入前，先锁定脚手架范围
- 识别哪些 RD 缺口会直接影响项目骨架，避免一边写代码一边改结构

## 输入前提

- 输入优先是 `flutter-prd-rd-writer` 生成的 RD
- 如果输入只有 PRD、脑图、需求摘要，先回退到 `flutter-prd-rd-writer`
- 如果 RD 同时覆盖多个独立业务域，先拆主 feature 与次级 feature，再初始化，不要一次生成模糊的大而全结构

## RD 章节到初始化动作的映射

### 1. `PRD 需求理解`

- 提取产品目标、目标用户、核心链路
- 决定示例首页、首批 feature、导航入口应该长什么样
- 判断这是 MVP、增长型还是复杂业务型项目

### 2. `前提假设与待确认项`

- 先读这一节，再决定是否立即初始化
- 只要某个未确认项会改变 feature 边界、鉴权方式、本地存储策略、路由守卫、离线能力，就先把它保留在初始化总结里
- 不要把这些缺口静默吞掉

### 3. `业务流程与页面范围`

- 转成首批 `lib/features/<feature>/presentation/pages`、`widgets` 与路由占位
- 页面范围不等于 feature 范围；按业务边界拆 feature，而不是按页面一页一个 feature

### 4. `总体技术架构`

- 决定 `lib/app`、`lib/core`、`lib/shared`、`lib/features` 的职责
- 决定是否需要多环境、远程配置、统一错误处理、会话刷新、埋点入口

### 5. `Flutter 客户端详细技术方案`

- 决定 app bootstrap、状态组织、网络层、序列化、本地缓存、错误模型的脚手架形态
- 这一节直接驱动 `pubspec.yaml`、`core/` 基础设施和 feature 模板

### 6. `第三方包选型与最佳搭档`

- 与 `flutter-project-guardrails` 对齐
- 先锁定必须使用的基础包，再补按能力触发的增量包
- 不要把 RD 里的包建议和 guardrails 混成两套体系

### 7. `后端协作与接口契约要求`

- 决定 `retrofit` 接口、DTO 目录、鉴权拦截器、分页模型、时区处理占位
- 如果接口契约还不稳定，只搭框架，不编造最终字段

### 8. `数据与安全方案`

- 决定 `flutter_secure_storage`、`shared_preferences`、离线数据库是否进首版脚手架
- 决定敏感信息脱敏、日志边界、缓存生命周期的基线

### 9. `埋点、监控与运营支持`

- 决定是否立即接入 `firebase_analytics`、`firebase_remote_config`、`firebase_crashlytics` 等能力
- 如果 RD 只要求预留能力，可以先搭抽象层和配置入口

### 10. `测试与质量保障`

- 至少决定 `flutter analyze`、`flutter test`、`build_runner` 是否进入初始化验证
- 如果 RD 要求契约测试、集成测试或 golden，先建目录和最小入口

### 11. `实施计划与里程碑`

- 提取首批 feature 优先级，写进生成后的 `flutter-dev`
- 提取并行开发边界、阻塞项和默认开发命令
- 把项目级规则和通用 guardrails 分开，避免后续开发每次都回看整份 RD

## 初始化前必抽取的信息

- 项目名、包名、平台范围
- 首批业务 feature 列表
- 鉴权模型与访客/登录态差异
- 网络协议类型与错误模型
- 本地存储与离线要求
- 远程配置、埋点、崩溃监控是否首期接入
- 多环境 / flavor / 构建配置要求

## 需要先停下来的高风险缺口

- 还没确认是否有登录、多角色、多租户
- 还没确认是否要离线数据库
- 还没确认是否有媒体上传、推送、实时通信
- 还没确认是否要灰度、远程开关、AB 实验
- 还没确认隐私、合规、审计边界

出现这些缺口时，先输出 `前提假设` 与 `待确认项`，再继续初始化。
