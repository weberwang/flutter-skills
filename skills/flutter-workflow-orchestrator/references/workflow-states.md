# Workflow States

Use this reference when classifying `current_stage`, module maturity, or allowed next moves.

Use one state per module:

| State | Meaning | Allowed Next Move |
| --- | --- | --- |
| `requirements_brainstorming` | Raw requirements, a one-line feature idea, or a partially specified request exists, but no PRD artifact is ready. The orchestrator must brainstorm goals, users, scope, risks, assumptions, and unresolved questions before generating the PRD. | internal requirements-to-PRD flow, then `flutter-prd-rd-writer` when ambiguities are resolved |
| `prd_ready` | PRD or feature brief exists, the question ledger is resolved strongly enough, and the PRD completeness gate has passed | `flutter-prd-rd-writer` for global technical baseline generation |
| `technical_baseline_ready` | Global architecture, package stack, backend collaboration, and delivery assumptions exist, but the target design-device preset plus base resolution, the Product Design brief, and the frozen shared visual direction are not yet ready. | freeze design-device preset and base resolution, then `@product-design` `get-context` |
| `shared_design_direction` | The target design-device preset and base resolution are frozen for this design cycle, `@product-design` has confirmed the design brief or is actively resolving it from the PRD and technical baseline, and the workflow is ready to turn approved reference images into the first reviewable representative sketch baseline through local `$imagegen`. When richer commercial visual evidence is useful, this stage may also open the pre-direction Creative Production branch before the final direction is locked. | public-shell confirmation, then representative sketch generation, plus optional pre-direction Creative Production exploration |
| `design_recommendation_ready` | `@product-design` has produced one or more reference-image-grounded representative directions, including one primary recommendation when the run is in manual mode, and the workflow has rendered a representative local sketch for review. The workflow is waiting for human confirmation or revision selection before calling `gpt-image-2-generator` for the representative final effect image. In `--auto`, the orchestrator may adopt the primary recommendation directly when no blocker remains. In `--full-auto`, that same primary recommendation may also satisfy the final-direction confirmation gate when no competing approved direction remains unresolved. | human confirmation or recommendation revision, then representative final effect-image generation |
| `product_direction_confirmed` | The user has explicitly confirmed the representative sketch as the final product design direction baseline, and the representative final effect image has been generated through `gpt-image-2-generator`. | output `DESIGN.md` |
| `design_md_ready` | The confirmed final design direction has been written into `DESIGN.md`-ready downstream inputs and the workflow is ready to drive HTML interactive prototype design-source work. Asset-oriented requests may also open the post-direction Creative Production asset-production branch from here without weakening the product-design freeze rules. | optional post-direction Creative Production branch, `html_design_source_ready` |
| `global_effect_image_pilot_ready` | One required representative local sketch exists for the page that best represents the product's overall visual system, usually from the approved reference-image plus Product Design flow, and the workflow is waiting for user confirmation or revision feedback before generating the representative final effect image and the remaining required page images. | user confirmation, then representative final effect-image generation and required remaining page-effect generation |
| `global_effect_images_ready` | Module-scoped or shared light-mode effect images exist where the active route requires them and are approved as visual evidence for downstream design-source work, freeze, or human visual inspection. Creative Production outputs may also exist as separate asset evidence, but they do not replace the selected HTML interactive prototype design source. | `html_design_source_ready` or `flutter-design-freeze-gate` |
| `html_design_source_ready` | A shared HTML interactive prototype has generated or validated the prototype-derived shared theme/public-shell design-source packet from the confirmed `DESIGN.md` plus the confirmed representative effect-image baseline. This global packet must not contain final module page implementation code. | `design-preview-to-global-guidelines` or `flutter-design-freeze-gate` |
| `global_guidelines_frozen` | The confirmed `DESIGN.md`, the selected shared HTML interactive prototype output, and the approved representative effect-image baseline have been converted into frozen shared guidance and explicit global public component decisions. No module page design is included at this stage. This is the first internal half of the shared freeze gate rather than a separate user-facing destination. | `flutter-design-freeze-gate` |
| `design_freeze_ready` | The shared theme/public-shell design packet, `DESIGN.md`, the selected global HTML interactive prototype artifacts, and the confirmed representative effect-image baseline are ready for approval before any module-related workflow may begin. The shared/global freeze covers theme and public shell only, not page design. Once this shared freeze gate passes, the workflow should prefer early project initialization plus bootstrap preparation and may also begin module implementation-document generation. | `flutter-init`, bootstrap preparation, or `flutter-rd-module-splitter` |
| `module_impl_docs_ready` | The shared design freeze is already complete, and the active module's detailed task implementation document `impl.md` has been generated under the frozen shared design and interaction principles. That `impl.md` fixes the module function, key states, task path, and non-display behavior boundaries strongly enough to drive the module effect-image generation and later module HTML interactive prototype generation. | module effect-image generation for the active module |
| `module_design_frozen` | The active module's implementation-final `impl.md`, module effect image, selected prototype-derived HTML design-source packet, high-fidelity visual contract, and any additional approved visual evidence are frozen and confirmed. | `flutter-design-source-control` or `flutter-uiux-to-architecture` |
| `impl_rd_ready` | The active module's implementation RD is implementation-final or landed, references the frozen prototype-derived HTML design-source packet and global technical baseline, and is confirmed as implementable. | `flutter-uiux-to-architecture` |
| `architecture_ready` | Tokens, assets, components, screen plan, non-native visual asset strategy, and scaffold contract exist. | implementation entry once `bootstrap_code_ready` and other implementation prerequisites already exist |
| `project_initialized` | `flutter-init` has only created the directory skeleton and sibling `skills/flutter-dev/` container, but has not started bootstrap code, shared wiring, or any feature implementation code. Unless explicitly overridden, the workflow should already treat `fvm` as the default Flutter environment contract from this point forward. | bootstrap code generation |
| `bootstrap_code_ready` | Global bootstrap code exists: app entry, root app shell, shared wiring, and core startup scaffolding are landed, but feature-module implementation has not started yet. Flutter and Dart execution guidance should still default to `fvm` unless the user explicitly changed the environment contract. | `@superpowers` `Spec` for implementation entry |
| `implementing` | Implementation entry begins only after bootstrap code is ready, the active module's selected HTML interactive prototype design source is frozen, and the module `impl.md` is executable. The module must first pass through `@superpowers` `Spec`, then `@superpowers` `Plan`, and only then execute code work. The module-level implementation loop is fixed and serial: confirm `impl.md` as the functional contract, consume the frozen module effect image and module HTML interactive prototype, freeze a page-level `display_restoration_blueprint`, land non-display-layer code against that blueprint, and restore the display layer from that same blueprint last. The orchestrator processes modules one by one rather than opening parallel module lanes; display-layer work must inspect the corresponding page image, module effect image, HTML prototype, `display_restoration_blueprint`, and frozen design evidence first when such evidence exists. When a source visual target and implementation screenshots both exist, `@product-design` `design-qa` may run as a blocking helper, but final app-page acceptance must happen only at the end and not inside `design-qa`. human visual inspection remains the final acceptance owner. | `@superpowers` `Spec` -> `@superpowers` `Plan` -> serial module implementation execution -> optional blocking `@product-design` `design-qa` -> final human visual inspection |
| `module_done` | Shared freeze is confirmed, module implementation docs are landed, the selected HTML interactive design source is frozen, code is landed, and the user has accepted the final human visual inspection result. human visual inspection remains the final acceptance owner, and final app-page acceptance must happen only at the end of the workflow. | maintain index only |
