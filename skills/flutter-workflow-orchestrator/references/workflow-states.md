# Workflow States

Use this reference when classifying `current_stage`, module maturity, or allowed next moves.

Use one state per module:

| State | Meaning | Allowed Next Move |
| --- | --- | --- |
| `idea_sketch_brainstorming` | The input is still a rough idea, reference bundle, or abstract product direction that is not concrete enough to reason about through pages yet. The orchestrator must brainstorm the core product path, generate `1-3` key-page sketches, confirm product direction plus page structure plus broad visual style, and write `docs/project/idea-sketch-brief.md` before PRD work may begin. | sketch confirmation, brief artifact write, then `requirements_brainstorming` |
| `requirements_brainstorming` | The workflow has a confirmed `docs/project/idea-sketch-brief.md` or an already-concrete feature request, but no PRD artifact is ready yet. The orchestrator must complete the requirement-space brainstorming needed for PRD quality: goals, users, scope, rules, risks, assumptions, and unresolved questions. | internal requirements-to-PRD flow, then `flutter-prd-rd-writer` when ambiguities are resolved |
| `prd_ready` | PRD or feature brief exists, the question ledger is resolved strongly enough, and the PRD completeness gate has passed | `flutter-prd-rd-writer` for global technical baseline generation |
| `technical_baseline_ready` | Global architecture, package stack, backend collaboration, and delivery assumptions exist, but the target design-device preset plus base resolution, the Product Design brief, and the frozen shared visual direction are not yet ready. | freeze design-device preset and base resolution, then `@product-design` `get-context` |
| `shared_design_direction` | The target design-device preset and base resolution are frozen for this design cycle, `@product-design` has confirmed the design brief or is actively resolving it from the PRD and technical baseline, and the workflow is ready to turn approved reference images into the first reviewable representative sketch baseline through local `$imagegen`. When richer commercial visual evidence is useful, this stage may also open the pre-direction Creative Production branch before the final direction is locked. | public-shell confirmation, then representative sketch generation, plus optional pre-direction Creative Production exploration |
| `design_recommendation_ready` | `@product-design` has produced one or more reference-image-grounded representative directions, including one primary recommendation when the run is in manual mode, and the workflow has rendered a representative local sketch for review. The workflow is waiting for human confirmation or revision selection before calling `gpt-image-2-generator` for the representative final effect image. In `--auto`, the orchestrator may adopt the primary recommendation directly when no blocker remains. In `--full-auto`, that same primary recommendation may also satisfy the final-direction confirmation gate when no competing approved direction remains unresolved. | human confirmation or recommendation revision, then representative final effect-image generation |
| `product_direction_confirmed` | The user has explicitly confirmed the representative sketch as the final product design direction baseline, and the representative final effect image has been generated through `gpt-image-2-generator`. | output `DESIGN.md` |
| `design_md_ready` | The confirmed final design direction has been written into `DESIGN.md`-ready downstream inputs and the workflow is ready to build the mandatory shared Pencil design source before deciding post-review bitmap generation. Asset-oriented requests may also open the post-direction Creative Production asset-production branch from here without weakening the product-design freeze rules. | optional post-direction Creative Production branch, `pencil_design_source_ready`, then `shared_asset_resources_ready` when required |
| `global_effect_image_pilot_ready` | One required representative local sketch exists for the page that best represents the product's overall visual system, usually from the approved reference-image plus Product Design flow, and the workflow is waiting for user confirmation or revision feedback before generating the representative final effect image and the remaining required page images. | user confirmation, then representative final effect-image generation and required remaining page-effect generation |
| `global_effect_images_ready` | Shared light-mode effect images exist where the shared route requires them and are approved as visual evidence for downstream design-source work, Pencil review, post-review single-asset generation, freeze, or human visual inspection. That evidence now also includes the matching UI-only transparent atlas artifacts plus manifest for each required effect-image page. Module-scoped effect images may also reuse this state name, but only after the active module has already entered the implementation-stage visual restoration loop. Creative Production outputs may also exist as separate asset evidence, but they do not replace the selected structured design source. | `pencil_design_source_ready`, `shared_asset_resources_ready`, `module_asset_resources_ready`, `display_evidence_pack_ready`, or `flutter-design-freeze-gate` |
| `pencil_design_source_ready` | The current shared scope, or the active module after it has already entered implementation, has already routed through `effect-image-to-pencil-design`, built the mandatory high-fidelity Pencil design source, compared it against the approved effect image, completed one confirmed repair pass, and passed human acceptance for the Pencil file. This review result is now the basis for deciding whether any new bitmap generation is still required. | `shared_asset_resources_ready`, `module_asset_resources_ready`, `global_guidelines_frozen`, `module_design_frozen`, or `flutter-design-freeze-gate` depending on scope |
| `shared_asset_resources_ready` | After the shared Pencil design review was accepted, the shared scope processed each target shared page one by one: the workflow identified that page's remaining new bitmap assets, resolved or queued reuse questions, waited for explicit confirmation of that page's post-review bitmap list, generated each approved resource independently through `$imagegen`, and saved the final approved asset files that downstream structured design sources must reference directly. | `global_guidelines_frozen`, `flutter-design-freeze-gate`, or `module_design_frozen` depending on scope |
| `global_guidelines_frozen` | The confirmed `DESIGN.md`, the selected shared structured design-source packet, and the approved representative effect-image baseline have been converted into frozen shared guidance and explicit global public component decisions. No module page design is included at this stage. This is the first internal half of the shared freeze gate rather than a separate user-facing destination. | `flutter-design-freeze-gate` |
| `design_freeze_ready` | The shared theme/public-shell design packet, `DESIGN.md`, the frozen shared Pencil design source, the selected additional structured design-source artifacts, any approved shared generated asset files that remained necessary after review, and the confirmed representative effect-image baseline are ready for approval before any module-related workflow may begin. The shared/global freeze covers theme and public shell only, not page design. Once this shared freeze gate passes, the workflow should prefer early project initialization plus bootstrap preparation and may also begin module implementation-document generation. | `flutter-init`, bootstrap preparation, or `flutter-rd-module-splitter` |
| `module_impl_docs_ready` | The shared design freeze is already complete, and the active module's detailed task implementation document `impl.md` has been generated under the frozen shared design and interaction principles. That `impl.md` fixes the module function, key states, task path, and non-display behavior boundaries strongly enough to freeze the module contract and later drive the implementation-stage visual restoration loop, including module effect-image generation, display-evidence-pack confirmation, module Pencil review, and any post-review module image-asset generation when they are truly required. | `module_design_frozen` |
| `module_asset_resources_ready` | This is an implementation-stage milestone only. After the active module has already entered the visual restoration loop and the module Pencil design review was accepted, the workflow processed each target module page one by one: it identified that page's remaining new bitmap assets, resolved or queued reuse questions through the global asset registry, waited for explicit confirmation of that page's post-review bitmap list, generated each approved resource independently through `$imagegen`, and saved the final approved asset files that the module's structured design source and later code must reference directly. | `pencil_restoration_ready` or continue `implementing` depending on scope |
| `display_evidence_pack_ready` | The active scope now has enough readable visual evidence for fidelity-critical regions: main preview, required detail views, key state views, scroll evidence, and overlay evidence where needed. The workflow has confirmed that later freeze, architecture, and display restoration do not need to guess missing structure or state language. | `flutter-design-freeze-gate` or `flutter-uiux-to-architecture` depending on scope |
| `module_design_frozen` | The active module's implementation-final `impl.md`, inherited frozen shared design constraints, selected structured design-source packet, explicit module scope decisions, and high-fidelity restoration target are frozen strongly enough to enter implementation. Module effect images, module Pencil execution, and module bitmap generation do not block this pre-implementation freeze anymore; they belong to the later implementation-stage visual restoration loop. | `flutter-design-source-control`, `impl_rd_ready`, or `pencil_restoration_ready` |
| `impl_rd_ready` | The active module's implementation RD is implementation-final or landed, references the frozen structured design-source packet and global technical baseline, and is confirmed as implementable. | `flutter-uiux-to-architecture` |
| `pencil_restoration_ready` | The active module has already routed through `pencil-design-to-flutter-restoration` and now has a restoration contract that explicitly classifies Flutter standard-library fit, third-party plugin fit, and approved bitmap-asset usage for the frozen Pencil design source. | `flutter-uiux-to-architecture` |
| `architecture_ready` | Tokens, assets, components, screen plan, non-native visual asset strategy, and scaffold contract exist, but the workflow may still need to confirm that the page-level display restoration contract is explicit enough for implementation. | `display_restoration_blueprint_ready` |
| `display_restoration_blueprint_ready` | The active module now has a concrete display-layer decision table and a page-level `display_restoration_blueprint` covering region ownership, scroll and sticky rules, state slots, asset bindings, and fidelity locks. This is the implementation-facing display contract and is mandatory before code execution begins. | implementation entry once `bootstrap_code_ready` and other implementation prerequisites already exist |
| `project_initialized` | `flutter-init` has only created the directory skeleton and sibling `skills/flutter-dev/` container, but has not started bootstrap code, shared wiring, or any feature implementation code. Unless explicitly overridden, the workflow should already treat `fvm` as the default Flutter environment contract from this point forward. | bootstrap code generation |
| `bootstrap_code_ready` | Global bootstrap code exists: app entry, root app shell, shared wiring, and core startup scaffolding are landed, but feature-module implementation has not started yet. Flutter and Dart execution guidance should still default to `fvm` unless the user explicitly changed the environment contract. | `@superpowers` `Spec` for implementation entry |
| `implementing` | Implementation entry begins only after bootstrap code is ready, `display_restoration_blueprint_ready` is already confirmed, and the module `impl.md` is executable. The module must first pass through `@superpowers` `Spec`, then `@superpowers` `Plan`, and only then execute code work. The module-level implementation loop is fixed and serial: confirm `impl.md` as the functional contract, generate or refresh the module effect image under frozen shared constraints together with its matching UI-only transparent atlas artifacts, confirm the display evidence pack, execute module Pencil review plus one focused repair pass when needed, generate only the still-required module bitmap assets, honor the frozen restoration contract plus `display_restoration_blueprint`, land non-display-layer code against that blueprint, and restore the display layer from that same blueprint last. The orchestrator processes modules one by one rather than opening parallel module lanes; display-layer work must inspect the corresponding page image, implementation-stage module effect image, frozen Pencil design, atlas manifest, `display_restoration_blueprint`, and frozen design evidence first when such evidence exists. When implementation screenshots exist, the workflow should route through render-vs-frozen-design QA before final helper review and human acceptance. | `@superpowers` `Spec` -> `@superpowers` `Plan` -> serial module implementation execution -> `flutter_render_vs_frozen_design_qa` -> optional blocking `@product-design` `design-qa` -> final human visual inspection |
| `flutter_render_vs_frozen_design_qa` | The active module's rendered Flutter result has been compared against the frozen design source, approved effect images, approved generated asset files, and `display_restoration_blueprint`, and the workflow has an explicit render-level fidelity QA result before final human acceptance. | optional blocking `@product-design` `design-qa`, then final human visual inspection |
| `module_done` | Shared freeze is confirmed, module implementation docs are landed, the selected structured design source is frozen, `display_restoration_blueprint_ready` is confirmed, code is landed, render-vs-frozen-design QA has already run, and the user has accepted the final human visual inspection result. human visual inspection remains the final acceptance owner, and final app-page acceptance must happen only at the end of the workflow. | maintain index only |
