# Workflow States

Use this reference when classifying `current_stage`, module maturity, or allowed next moves.

Use one state per module:

| State | Meaning | Allowed Next Move |
| --- | --- | --- |
| `requirements_brainstorming` | Raw requirements, a one-line feature idea, or a partially specified request exists, but no PRD artifact is ready. The orchestrator must brainstorm goals, users, scope, risks, assumptions, and unresolved questions before generating the PRD. | internal requirements-to-PRD flow, then `flutter-prd-rd-writer` when ambiguities are resolved |
| `prd_ready` | PRD or feature brief exists, the question ledger is resolved strongly enough, and the PRD completeness gate has passed | `flutter-prd-rd-writer` for global technical baseline generation |
| `technical_baseline_ready` | Global architecture, package stack, backend collaboration, and delivery assumptions exist, but the target design-device preset plus base resolution, the Product Design brief, and the frozen shared visual direction are not yet ready. | freeze design-device preset and base resolution, then `@product-design` `get-context` |
| `shared_taste_direction` | The target design-device preset and base resolution are frozen for this design cycle, `@product-design` has confirmed the design brief or is actively resolving it from the PRD and technical baseline, and the workflow is converging shared public-shell rules, product posture, information density, typography hierarchy, contrast posture, CTA model, motion role, platform baseline, and shared interaction principles. When richer commercial visual evidence is useful, this stage may also open the pre-direction Creative Production branch before the final direction is locked. | public-shell confirmation, then `@product-design` visual-direction recommendation, plus optional pre-direction Creative Production exploration |
| `design_recommendation_ready` | `@product-design` has produced one or more recommended global design directions grounded in the confirmed brief and public-shell decisions, including a small style recommendation set with one primary recommendation when the run is in manual mode, and the workflow is waiting for human confirmation or revision selection. In `--auto`, the orchestrator may adopt the primary recommendation directly when no blocker remains. | human confirmation or recommendation revision, or auto-adopt the primary recommendation in `--auto` |
| `product_direction_confirmed` | The user has explicitly confirmed the final product design direction after reviewing the `@product-design` brief output or recommendation package. | output `DESIGN.md` |
| `design_md_ready` | The confirmed final design direction has been written into `DESIGN.md` and is ready to drive downstream structured design-source work. Effect images are optional supplemental evidence, not a mandatory gate. Asset-oriented requests may also open the post-direction Creative Production asset-production branch from here without weakening the product-design freeze rules. | optional post-direction Creative Production branch, optional effect-image generation, `stitch_design_source_ready`, or `pencil_design_source_ready` |
| `global_effect_image_pilot_ready` | One optional representative light-mode effect-image baseline exists for the page that best represents the product's overall visual system, usually from the approved Product Design visual target, and the workflow is waiting for user confirmation or revision feedback before generating any additional optional page images. | user confirmation, then optional remaining page-effect generation |
| `global_effect_images_ready` | Optional page-addressable effect images exist and are approved as supplemental visual evidence for downstream design-source work, freeze, or human visual inspection. Creative Production outputs may also exist as separate asset evidence, but they do not replace the selected structured design-source adapter. | `stitch_design_source_ready`, `pencil_design_source_ready`, or `flutter-design-freeze-gate` |
| `stitch_design_source_ready` | Stitch has generated or validated the shared theme/public-shell structured design-source packet from the confirmed `DESIGN.md` plus any optional visual evidence, with frozen `stitch_project_id`. This global packet must not contain page designs. | `design-preview-to-global-guidelines` or `flutter-design-freeze-gate` |
| `pencil_design_source_ready` | Pencil has generated or validated the shared theme/public-shell structured design-source packet from the confirmed `DESIGN.md` plus any optional visual evidence, with the project-level Pencil source reference frozen. This global packet must not contain page designs. | `design-preview-to-global-guidelines` or `flutter-design-freeze-gate` |
| `global_guidelines_frozen` | The confirmed `DESIGN.md`, the selected shared theme/public-shell design-source output, and any approved optional visual evidence have been converted into frozen shared guidance and explicit global public component decisions. No module page design is included at this stage. This is the first internal half of the shared freeze gate rather than a separate user-facing destination. | `flutter-design-freeze-gate` |
| `design_freeze_ready` | The shared theme/public-shell design packet, `DESIGN.md`, the selected global structured design-source artifacts, and any required optional visual evidence are ready for approval before any module-related workflow may begin. The shared/global freeze covers theme and public shell only, not page design. Once this shared freeze gate passes, the workflow should prefer early project initialization plus bootstrap preparation and may also begin module implementation-document generation. | `flutter-init`, bootstrap preparation, or `flutter-rd-module-splitter` |
| `module_impl_docs_ready` | The shared design freeze is already complete, and the active module's detailed task implementation document `impl.md` has been generated under the frozen shared design and interaction principles. That `impl.md` fixes the module function, key states, task path, and non-display behavior boundaries strongly enough to drive the module page component design draft and later implementation readiness. | selected design-source generation or validation for the active module |
| `module_design_frozen` | The active module's implementation-final `impl.md`, module page component design draft, selected structured design-source packet, high-fidelity visual contract, optional visual evidence, component freeze, and state matrix are frozen and confirmed. | `flutter-design-source-control` or `flutter-uiux-to-architecture` |
| `impl_rd_ready` | The active module's implementation RD is implementation-final or landed, references the frozen structured design-source packet and global technical baseline, and is confirmed as implementable. | `flutter-uiux-to-architecture` |
| `architecture_ready` | Tokens, assets, components, screen plan, non-native visual asset strategy, and scaffold contract exist. | implementation entry once `bootstrap_code_ready` and other implementation prerequisites already exist |
| `project_initialized` | `flutter-init` has only created the directory skeleton and project-local `skills/flutter-dev/` container, but has not started bootstrap code, shared wiring, or any feature implementation code. | bootstrap code generation |
| `bootstrap_code_ready` | Global bootstrap code exists: app entry, root app shell, shared wiring, and core startup scaffolding are landed, but feature-module implementation has not started yet. | `@superpowers` `Spec` for implementation entry |
| `implementing` | Implementation entry begins only after bootstrap code is ready, the active module's selected structured design source is frozen, and the module `impl.md` is executable. The module must first pass through `@superpowers` `Spec`, then `@superpowers` `Plan`, and only then execute code work. The module-level implementation loop is fixed and serial: confirm `impl.md` as the functional contract, consume the frozen module page component design draft, land non-display-layer code first, and restore the display layer from that draft last. The orchestrator processes modules one by one rather than opening parallel module lanes; display-layer work must inspect the corresponding page image and frozen design evidence first when such evidence exists. When a source visual target and implementation screenshots both exist, `@product-design` `design-qa` becomes the blocking QA helper before the final human visual inspection handoff. | `@superpowers` `Spec` -> `@superpowers` `Plan` -> serial module implementation execution -> `@product-design` `design-qa` -> human visual inspection |
| `module_done` | Shared freeze is confirmed, module implementation docs are landed, the selected structured design source is frozen, code is landed, and the user has accepted the human visual inspection result. | maintain index only |
