# Workflow States

Use this reference when classifying `current_stage`, module maturity, or allowed next moves.

Use one state per module:

| State | Meaning | Allowed Next Move |
| --- | --- | --- |
| `requirements_brainstorming` | Raw requirements, a one-line feature idea, or a partially specified request exists, but no PRD artifact is ready. The orchestrator must brainstorm goals, users, scope, risks, assumptions, and unresolved questions before generating the PRD. | internal requirements-to-PRD flow, then `flutter-prd-rd-writer` when ambiguities are resolved |
| `prd_ready` | PRD or feature brief exists | `flutter-prd-rd-writer` for global technical baseline generation |
| `technical_baseline_ready` | Global architecture, package stack, backend collaboration, and delivery assumptions exist, but executable module `impl.md` documents have not yet been generated from that baseline. | `flutter-rd-module-splitter` |
| `module_impl_docs_ready` | Detailed modules, module index rows, module detail cards, global baseline references, module-level component skeletons, and executable module `impl.md` documents exist from one combined generation step. Each `impl.md` is already implementation-final enough to drive downstream visual design, module freeze, and implementation readiness; separate refinement documents are not required. | `flutter-taste-router` |
| `shared_taste_direction` | The global visual design direction has been brainstormed from the PRD, technical baseline, and executable module `impl.md` contracts: product posture, anti-template rules, information density, typography hierarchy, contrast posture, CTA model, motion role, and platform baseline are defined. At this point the workflow must choose how to confirm the final direction: direct human confirmation, or a subagent-generated recommendation package followed by human confirmation. | direct confirmation or subagent recommendation generation |
| `design_recommendation_ready` | A subagent has produced one or more recommended design directions grounded in the shared taste packet and public-shell decisions, and the workflow is waiting for human confirmation or revision selection. | human confirmation or recommendation revision |
| `product_direction_confirmed` | The user has explicitly confirmed the final product design direction after the global visual design brainstorming step, either directly or after reviewing a subagent recommendation package. | output `DESIGN.md` |
| `design_md_ready` | The confirmed final design direction has been written into `DESIGN.md` and is ready to drive downstream structured design-source work. Effect images are optional supplemental evidence, not a mandatory gate. | optional effect-image generation, `stitch_design_source_ready`, or `pencil_design_source_ready` |
| `global_effect_image_pilot_ready` | One optional representative light-mode effect image has been generated for the page that best represents the product's overall visual system, and the workflow is waiting for user confirmation or revision feedback before generating any additional optional page images. | user confirmation, then optional remaining page-effect generation |
| `global_effect_images_ready` | Optional page-addressable effect images exist and are approved as supplemental visual evidence for downstream design-source work, freeze, or human visual inspection. | `stitch_design_source_ready`, `pencil_design_source_ready`, or `flutter-design-freeze-gate` |
| `stitch_design_source_ready` | Stitch has generated or validated the structured design-source packet from the confirmed `DESIGN.md` plus any optional visual evidence, with frozen `stitch_project_id`. | `design-preview-to-global-guidelines` or `flutter-design-freeze-gate` |
| `pencil_design_source_ready` | Pencil has generated or validated the structured design-source packet from the confirmed `DESIGN.md` plus any optional visual evidence, with the project-level Pencil source reference frozen. | `design-preview-to-global-guidelines` or `flutter-design-freeze-gate` |
| `global_guidelines_frozen` | The confirmed `DESIGN.md`, the selected structured design-source output, and any approved optional visual evidence have been converted into frozen shared guidance and explicit global public component decisions. | `flutter-design-freeze-gate` |
| `design_freeze_ready` | The shared design packet, `DESIGN.md`, the selected structured design-source artifacts, and any required optional visual evidence are ready for approval before module design freeze. | `flutter-design-freeze-gate` |
| `module_design_frozen` | The active module's implementation-final `impl.md`, selected structured design-source packet, high-fidelity visual contract, optional visual evidence, component freeze, and state matrix are frozen and confirmed. | `flutter-design-source-control` or `flutter-uiux-to-architecture` |
| `impl_rd_ready` | The active module's implementation RD is implementation-final or landed, references the frozen structured design-source packet and global technical baseline, and is confirmed as implementable. | `flutter-uiux-to-architecture` |
| `architecture_ready` | Tokens, assets, components, screen plan, non-native visual asset strategy, and scaffold contract exist. | `flutter-init`, especially once the shared public baseline is clear enough to initialize the project before feature-module code |
| `project_initialized` | `flutter-init` has only created the directory skeleton and project-local `skills/flutter-dev/` container, but has not started bootstrap code, shared wiring, or any feature implementation code. | bootstrap code generation |
| `bootstrap_code_ready` | Global bootstrap code exists: app entry, root app shell, shared wiring, and core startup scaffolding are landed, but feature-module implementation has not started yet. | `@superpowers` `Spec` for implementation entry |
| `implementing` | Implementation entry begins only after bootstrap code is ready, the active module's selected structured design source is frozen, and the module `impl.md` is executable. The module must first pass through `@superpowers` `Spec`, then `@superpowers` `Plan`, and only then execute code work. Execution defaults to parallel implementation of independent work packages with explicit ownership boundaries; display-layer work must inspect the corresponding page image through `$image-to-code` first when such evidence exists. | `@superpowers` `Spec` -> `@superpowers` `Plan` -> parallel implementation execution -> human visual inspection |
| `module_done` | Shared freeze is confirmed, module implementation docs are landed, the selected structured design source is frozen, code is landed, and the user has accepted the human visual inspection result. | maintain index only |
