# Routing Rules

Use this reference before selecting any downstream skill, changing `current_stage`, or applying module maturity changes.

1. Start by ensuring workflow state is initialized for the current run. If runtime persistence is enabled, initialize the runtime artifact before any stage routing.
2. Before deciding or invoking anything downstream, derive one route lock and run the preflight gate. If either step fails, stop and record a blocker instead of letting the downstream skill reinterpret the workflow.
3. If `confirmation_status` is `pending_confirmation`, do not switch to the next process and do not apply queued status changes, unless `--auto` or `--full-auto` is active. In `--auto` mode, ordinary downstream confirmation gates are auto-confirmed until workflow completion or a blocker is reached. In `--full-auto`, deterministic human-facing gates may also be auto-confirmed when the control contracts prove there is exactly one supported default.
4. If the user explicitly confirms a pending transition or status update, promote the queued values, clear pending fields, and continue normal routing from the newly confirmed state.
5. If the user rejects a pending transition or status update, keep the current confirmed state, write the rejection reason, and route back to the skill that must revise artifacts.
6. If a specialist skill returns `blocked`, `not_executed`, or a receipt that fails route-lock validation, do not advance. Keep `current_stage` on the last confirmed stage, clear queued transitions, and record the blocker or route-drift reason.
7. If the input is raw requirements, a one-line feature idea, a partially specified request, or any request without a PRD artifact, set or keep `current_stage=requirements_brainstorming`, read `requirements-prd-flow.md`, brainstorm the requirement space, resolve all decision-blocking questions, and generate a PRD before advancing.
8. If brainstorming finds unresolved decision-blocking questions that cannot be answered from context or by low-risk defaults, do not call `flutter-prd-rd-writer`. Record the question ledger and stop with `required_inputs`.
9. Before promoting the run to `prd_ready`, read `prd-template.md` and `prd-completeness-gate.md`, ensure the PRD artifact exists on disk, ensure the question ledger is embedded or linked, and ensure the completeness outcome is `ready_for_prd_ready`.
10. If all PRD-blocking questions are resolved or explicitly defaulted and the completeness gate passes, queue `pending_next_stage=prd_ready` until confirmation unless `--auto` or `--full-auto` is active.
11. If the input is already a PRD, broad RD, or feature brief, use `flutter-prd-rd-writer` first to derive or refresh the global technical baseline, then queue or promote `technical_baseline_ready`.
12. If a global technical baseline exists but the target design-device preset and base resolution are not yet frozen for this design cycle, keep `current_stage=technical_baseline_ready`, record `required_inputs=design_device_preset_and_resolution`, and stop. In manual mode, offer at least these common iPhone presets unless the user already chose another viewport: `375 x 667 px`, `390 x 844 px`, `393 x 852 px`, and `430 x 932 px`. In `--auto` or `--full-auto`, if no frozen viewport exists yet, auto-freeze `390 x 844 px` as the design viewport and continue.
13. If a global technical baseline exists but the workflow lacks a confirmed Product Design brief for this request, first read `design-quality-guidance.md`, then route to `@product-design` `get-context`. Product design must be briefed before final product direction confirmation.
14. If a global technical baseline exists but the workflow lacks a confirmed structured Product Design clarification packet for later module splitting, keep the run on the shared-direction branch and route to `@product-design` scope clarification before any shared freeze completion or module splitting. That clarification must lock core user journeys, page families, critical states, interaction goals, explicit `platform_identifier`, platform-aware shell or navigation expectations, and per-surface information-density posture.
15. Do not generate module `impl.md` documents, module effect images, module HTML interactive prototypes, or module freeze artifacts before the shared/global design freeze has completed.
16. If the common public shell is not yet explicitly agreed with the user, set or keep `current_stage=shared_design_direction`, record `required_inputs=public_shell_confirmation`, and stop. In `--full-auto`, if the current shell packet has exactly one supported shared-shell contract and no competing shell option remains in scope, auto-confirm it and continue instead of stopping.
17. If the Product Design brief for this request has not yet been confirmed, set or keep `current_stage=shared_design_direction`, route to `@product-design` `get-context`, and stop until the brief is confirmed.
18. If the Product Design brief is confirmed but the workflow has not yet completed a Mobbin-backed inspiration pass for the current design cycle, keep `current_stage=shared_design_direction`, route to `@product-design` through a Mobbin-first shared/global style recommendation pass, and stop any direction recommendation, effect-image generation, or downstream freeze work until that pass is complete.
19. If the Product Design brief is confirmed and the Mobbin-backed inspiration pass is complete but no representative effect-image artifact exists yet, keep `current_stage=shared_design_direction`, route to `@product-design` visual ideation using the approved reference images plus a Mobbin-backed shared/global style recommendation pass, and queue `pending_next_stage=design_recommendation_ready` on success. Treat `https://mobbin.com/` only as inspiration evidence for this pass, not as the formal downstream design source.
20. If a representative effect-image recommendation artifact exists but the user has not yet confirmed a final direction, set or keep `current_stage=design_recommendation_ready`, record `required_inputs=product_design_final_direction_confirmation`, and stop. In manual mode, this confirmation step should expose exactly three recommended representative effect-image directions with one primary recommendation rather than a single hidden answer. In `--full-auto`, if one clear primary recommendation exists and no competing approved direction remains unresolved, auto-confirm that direction and continue.
21. If the workflow needs richer campaign, hero, mood-board, or commercial visual evidence before the representative effect image can be chosen confidently, keep the run on the shared-direction branch, read `creative-production-branch.md`, route into the pre-direction Creative Production branch, and return to Product Design confirmation with that evidence.
22. If the user already supplied a concrete approved visual source and confirms the generated representative effect image as the final direction after Product Design brief playback, allow that approval to satisfy the recommendation step and record `current_stage=product_direction_confirmed` or queue that stage according to the confirmation gate.
23. If the final product design direction is confirmed after the representative effect-image recommendation pass, record `current_stage=product_direction_confirmed` or queue that stage according to the confirmation gate. In `--auto`, if a strong primary recommended effect-image style exists among the three compared Mobbin-backed directions and no blocker remains, adopt that recommended style directly and continue without pausing for manual confirmation. In `--full-auto`, treat that same strong primary recommendation as sufficient final-direction confirmation when no competing approved direction remains unresolved.
24. After `product_direction_confirmed`, read both `design-quality-guidance.md` and `design-md-template.md`, then route to output a root-level `DESIGN.md` before entering HTML interactive prototype generation, shared freeze, module design freeze, architecture, or code.
25. When `DESIGN.md` is successfully written at the project root, queue or promote `current_stage=design_md_ready`.
26. Do not enter HTML interactive prototype generation, shared freeze, module design freeze, architecture, or code before the global visual design direction has been brainstormed, the common public shell has been agreed, the final product direction has been confirmed, and `DESIGN.md` exists.
27. Before HTML interactive prototype work, verify that `DESIGN.md` explicitly covers at least task hierarchy, first-screen CTA posture, interaction and feedback rules, responsive strategy, critical states and edge cases, content tone when those concerns are relevant to the product, the frozen base design viewport or device preset, and the platform-aware information-density posture for major surfaces. If that coverage is weak, route back to revise `DESIGN.md` before continuing.
28. If the request is primarily about campaign visuals, ads, mood boards, launch assets, social cards, hero directions, scenes, shots, logos, or other asset-oriented creative work, read `creative-production-branch.md` and route into `Creative Production:explore` once the brief is specific enough.
29. Before final product direction confirmation, treat the Creative Production branch only as direction input. After `design_md_ready`, treat it as the post-direction asset-production branch.
30. If the user already selected an asset family or asks directly for one of the focused Creative Production paths, skip the generic chooser and route directly to the scope-matched explorer: `moodboard-explorer`, `ads-explorer`, `offer-explorer`, `scene-explorer`, `shot-explorer`, or `logo-explorer`.
31. If the user needs publish-bound marketing finish, exact-size social outputs, or premium visual polish while preserving exact text, logos, charts, data, safe zones, and dimensions, route to `Creative Production:generative-polish` only after one direction or deterministic base is already selected.
32. Do not use the Creative Production branch to replace Product Design brief confirmation, product-surface direction selection, `DESIGN.md`, shared freeze, or HTML interactive prototype generation. Treat it as an asset branch, not as a shortcut around the app-design workflow.
33. Do not route broad marketing or campaign exploration straight to `gpt-image-2-generator` when the Creative Production branch is available and the user needs reviewable creative routes, boards, or publish-safe asset production. Use preview-image generation only for the app-preview branch or when the user explicitly wants lightweight page evidence instead of the richer asset workflow.
34. If a Creative Production run surfaces a materially different visual direction from the already confirmed product direction, stop downstream freeze or implementation promotion, route back to Product Design confirmation, and revise `DESIGN.md` before continuing.
35. Effect-image generation is mandatory in both manual mode and `--auto` / `--full-auto`. Route into representative effect-image generation immediately after the Product Design brief is confirmed, using the approved reference images and Product Design direction constraints as the source.
36. In manual mode, reuse the approved Product Design visual target and reference images as the representative light-mode effect-image baseline whenever possible. If no approved representative visual target exists yet, stop and route back to the Product Design recommendation branch instead of inventing a new visual baseline.
37. Before effect-image generation, verify that `gpt-image-2-generator` is available and authenticated. If access, credentials, or upstream visual generation capability is missing, stop and record the branch-specific blocker.
38. Before any shared/global or module-stage effect-image generation request is sent, read the frozen global design packet and extract the applicable visual constraints for the target page or module. Generated effect images must explicitly inherit the confirmed product design direction, the approved Product Design visual target, the frozen base design viewport, `DESIGN.md`, PRD constraints, public-shell decisions, art direction, visual system, CTA posture, palette direction, typography mood, component family cues, image-treatment posture, target page purpose, and any first-screen task priority that materially affects the page when those values exist. Keep the frozen design width unchanged, treat the frozen viewport height as the minimum effect-image height, and allow taller outputs only when the target page or region is intentionally scrollable.
39. In manual mode, save the representative effect image into the global effect-image directory under `docs/project/` using the corresponding page name, index its path in workflow state, set the workflow to wait for user confirmation or revision feedback, and do not generate the remaining required page effect images yet.
40. In manual mode, after the representative effect image is confirmed, freeze it as the current-cycle visual baseline, delete previously generated alternative effect images that are no longer part of the frozen packet, update indexes to keep only the active baseline, and record that alternative-direction browsing is closed. Do not reopen Product Design direction browsing or generate other-direction effect images inside the same design cycle.
41. In `--auto` or `--full-auto`, once the representative effect-image direction is chosen, freeze it as the current-cycle visual baseline and continue without inventing an extra all-page effect-image stage.
42. All newly generated shared or module-stage effect images must be light-mode visuals. Do not generate dark-mode freeze evidence unless the user explicitly requests a dark-only or dual-mode design cycle.
43. Before any global design freeze or module design freeze, keep `@product-design` as the primary design controller. If the frozen packet still needs normalization after the Product Design direction is already confirmed, normalize it from approved images and confirmed design artifacts, then route into the HTML interactive prototype flow.
44. Before HTML interactive prototype work, require a root-level `DESIGN.md`, the frozen global design packet, and the active module design package when module scope is involved.
45. The HTML interactive prototype flow may only execute, restore, validate, or serialize the already-frozen design guidance. It must not introduce hidden style defaults, tool-owned design decisions, or implementation-driven redesign. If a downstream prototype-derived output conflicts with the frozen global design, the active module design package, or `DESIGN.md`, treat the prototype output as wrong and route to revision instead of inheriting the conflict.
46. During shared/global freeze, the selected HTML interactive prototype flow may generate or validate only the shared theme system, shared public shell, shared public component families, and shared interaction principles. Do not ask it to generate final module page implementation code before the shared/global design freeze has completed.
47. If the confirmed `DESIGN.md`, the selected shared theme/public-shell HTML interactive prototype output, and the approved required effect-image evidence must become a reusable shared design-source contract with fixed global guidance, use `design-preview-to-global-guidelines`, then record `global_guidelines_frozen` as `pending_next_stage` until the user confirms.
50. Before `design_freeze_ready` and the shared/global design freeze complete, do not advance into any module-related workflow. Module `impl.md` generation, module effect-image generation, module HTML interactive prototype generation, module freeze, module architecture, and module code all stay behind the shared freeze gate.
50. If shared design has a candidate direction, confirmed `DESIGN.md`, and a selected prototype-derived design-source packet but no explicit freeze decision, use `flutter-design-freeze-gate` directly. Before freezing, explicitly verify that the selected design-source output still preserves the approved style direction, theme system, public shell contract, shared public component families, task hierarchy, interaction feedback behavior, responsive strategy, and critical-state language.
51. If `flutter-design-freeze-gate` finds hierarchy, task guidance, typography, contrast, CTA clarity, state coverage, feedback consistency, responsive strategy, or content-tone coverage still weak, do not route to freeze. Allow one shared revision pass through the approved image-backed design-source flow, `DESIGN.md` refinement, plus required effect-image regeneration when needed, then stop unless the user explicitly restarts the design cycle.
52. Once the shared/global design freeze is complete and the shared bootstrap-critical baseline is explicit enough, prefer `flutter-init` immediately instead of waiting for later module architecture milestones.
53. If the shared bootstrap-critical baseline is ready and the target project has not been scaffolded yet or the workspace does not contain sibling `skills/flutter-dev/`, use `flutter-init` before feature-module implementation begins.
54. `flutter-init` must stop at initialization boundaries. It may create the directory skeleton and sibling `skills/flutter-dev/` container, but it must not implement bootstrap code, shared wiring, feature code, page code, or module-specific business logic. Unless the user explicitly overrides the environment contract, route initialization and later command guidance under the assumption that Flutter is managed through `fvm`.
55. If `flutter-init` has completed and the directory skeleton plus sibling `skills/flutter-dev/` exist, record `project_initialized` as `pending_next_stage`.
56. After `project_initialized` is confirmed, route to the separate bootstrap code stage.
57. `project_initialized` is directory-only. Do not treat it as proof that bootstrap code, shared wiring, or any feature implementation already exists.
58. If bootstrap code has landed and the required global public baseline is actually present on disk, record or queue `bootstrap_code_ready`.
59. If the shared/global design freeze is complete but detailed modules and executable module `impl.md` paths do not exist yet, first verify that the confirmed Product Design clarification packet for module splitting is present. If it is missing, route back to `@product-design` scope clarification and stop. Otherwise use `flutter-rd-module-splitter` to create module index rows and implementation-final `impl.md` documents in one pass. Those module documents must explicitly inherit the frozen shared design direction, shared interaction principles, shared public shell decisions, shared component constraints, platform-specific interaction expectations, and approved information-density posture.
60. If multiple feature modules depend on one shared route host, root redirect layer, tab shell, or shell-level state, allow `flutter-rd-module-splitter` to split an `app-shell` or `root-shell` module first instead of burying that responsibility inside feature modules, but still only after the shared/global design freeze is complete.
61. If a selected module lacks an executable `docs/project/modules/<module>/<module>.impl.md`, route back to `flutter-rd-module-splitter` for a scope-matched regeneration instead of opening a separate refinement stage.
62. Module implementation preparation consumes the executable `impl.md` produced by `flutter-rd-module-splitter`. Treat that `impl.md` as a detailed module task implementation document constrained by the frozen shared design and interaction principles. Do not require or create `docs/project/modules/<module>/<module>.ui-ux.md`.
63. When `flutter-rd-module-splitter` produces executable module `impl.md` documents, keep `current_stage` at the last confirmed stage, queue `pending_next_stage=module_impl_docs_ready`, queue each generated module's `impl_status=implementation_final`, and wait for confirmation before the active module's effect-image generation. In `--auto` or `--full-auto` mode, auto-apply those queued updates, select the first unfinished module in the confirmed serial module order, and continue directly into that active module's effect-image step.
64. In module freeze, the frozen prototype-derived design-source packet is required. The module HTML interactive prototype must be derived only after the active module's `impl.md` has already fixed the module function, key states, main task path, and non-display behavior boundaries under the frozen shared design and interaction principles, and after the module effect image has already been confirmed strongly enough for display work. If the active module lacks that packet or that function contract is still vague, stop and route back to the correct upstream regeneration path.
65. If the active module needs an effect image or implementation-facing design-source packet, first verify that the active module's `impl.md` already fixes the module function strongly enough for display work, including what must remain visible immediately on each important surface versus what must be deferred, collapsed, paged, or moved to another page for the real target platform. Then generate the module effect image, and only after that route to the selected HTML interactive prototype generation or validation path. Module-scoped page prototype generation is allowed only here, after shared/global freeze, never inside the shared/global freeze packet itself. Then route to `flutter-design-freeze-gate` before queueing `module_design_frozen`. Treat the module's high-fidelity visual contract as the first freeze question, and require the design packet to consider the real target platform plus the intended premium quality bar before freeze.
66. During executable module document generation and module design-freeze preparation, do not generate extra module real-device effect images by default beyond the required module effect-image step. Generate additional module image evidence only when the current route explicitly requires it.
67. When module-stage effect images are generated, save them into the global effect-image directory under `docs/project/`, then write the generated global effect-image paths into `global-design-guidelines.md`.
68. If `flutter-design-freeze-gate` finds the active-module design package not yet strong enough to freeze, keep the workflow on the active module, update the selected design-source packet, module `impl.md`, the required module effect image, and any additional approved visual evidence exactly once, and stop without another automatic freeze pass unless the user explicitly restarts a design cycle.
69. When `flutter-design-freeze-gate` approves the active module design source, first verify its receipt includes `high_fidelity_freeze_status=passed` or an explicit approved reduction. Before marking it frozen, delete obsolete pre-freeze module effect images that are no longer selected for this frozen packet and clear their workflow references. Do not immediately mark it frozen. Queue `pending_next_stage=module_design_frozen`, queue `design_source_status=frozen`, and if `impl.md` references the frozen prototype-derived design-source packet, also queue `impl_status=landed`.
70. If a frozen prototype-derived design source is about to be consumed by implementation RD or code, use `flutter-design-source-control`.
71. If frozen prototype-derived design-source output, theme values, component rules, required effect-image evidence, and any additional approved visual evidence must become Flutter-facing tokens, assets, components, screen architecture, and non-native visual fallback decisions, route to `flutter-uiux-to-architecture`.
72. Before architecture planning, implementation-readiness confirmation, human visual inspection, or code implementation, verify that `platform_identifier` is explicit enough for the primary runtime and validation platform. Do not treat `platform_baseline` such as `ios_hig` as if it already meant a runnable primary platform. Module-level design must already have considered this concrete primary platform before architecture inherits the packet.
73. When validation or debugging must run on the primary platform, select only one execution target for that platform. If exactly one eligible device exists, use it. If multiple eligible devices exist, stop and request explicit device selection. In `--full-auto`, if multiple eligible devices exist but exactly one of them is already booted, use that booted device as the deterministic default; otherwise stop. If no eligible device exists and the primary platform supports emulator or simulator startup, start one and use that fallback for validation.
74. Do not move a module into `implementing` until `technical_baseline_ready`, `design_freeze_ready`, `module_impl_docs_ready`, `module_design_frozen`, `impl_rd_ready`, and `bootstrap_code_ready` exist for the module, confirmed maturity is at least `impl_status=landed` and `design_source_status=frozen`, the frozen prototype-derived design-source packet exists, and the required global public code baseline is already landed.
75. The required global public code baseline before module implementation must include app bootstrap and environment initialization, root router or route host plus root redirect policy, global dependency injection or provider scope entry, local storage baseline and persistence wiring, global error mapping and logging baseline, the shared theme or design-token baseline required by the frozen prototype-derived design source, and the shared shell layer when feature modules depend on an `app-shell` or `root-shell`. Unless the user explicitly chooses otherwise, all Flutter and Dart command examples, execution steps, and validation guidance should default to the `fvm` toolchain contract.
76. Add network baseline and API client wiring to the global public code baseline only when the project or target modules actually require remote data, API access, upload, sync, or other network capabilities.
77. If `--auto` or `--full-auto` is active, do not stop for `project_initialized`, `bootstrap_code_ready`, `implementation_final`, `module_design_frozen`, `impl_rd_ready`, `architecture_ready`, `implementing`, or other downstream confirmation gates. Keep advancing until all target modules are fully implemented or a blocker appears, but still process modules strictly one by one.
78. If `--auto` or `--full-auto` is active and all target modules are fully implemented, stop here and surface workflow completion.
79. If implementation work should begin, first route through `@superpowers` to produce `Spec` for the active module. Do not begin execution before that `Spec` exists.
80. After `Spec` is confirmed, route through `@superpowers` again to produce `Plan` for the active module. Do not begin code execution before that `Plan` exists.
81. Only after both `Spec` and `Plan` exist may implementation execution begin through explicit `@superpowers` together with sibling `flutter-dev` and `flutter-project-guardrails`.
82. Implementation execution for module work is serial by default. The orchestrator should keep one active module at a time and land that module's internal steps in order unless the user explicitly changes the control model and the workflow contract is updated.
82. During module implementation, the module loop order is fixed: `impl.md` defines the functional contract, the frozen module effect image plus module HTML interactive prototype define the target UI, `flutter-uiux-to-architecture` must then materialize a page-level `display_restoration_blueprint`, non-display-layer code lands against that blueprint, and display-layer restoration lands after the relevant non-display contracts are stable enough.
83. The default module landing order is: confirm the frozen `impl.md` contract first, confirm the frozen module effect image plus module HTML interactive prototype second, freeze the `display_restoration_blueprint` third, land non-display-layer code fourth, and restore the display layer from that same blueprint fifth.
84. For ordinary business-page modules, prefer `impl contract -> display_restoration_blueprint -> non-display implementation -> display-layer restoration`. For infrastructure-heavy modules, it is allowed to keep the display step minimal or absent, but if a display surface exists it should still land after the non-display behavior is stable enough to support it.
85. Before display-layer restoration begins, run a display-layer readiness preflight and leave an explicit receipt. Verify that the page has enough readable visual evidence when needed, that the selected HTML interactive prototype design source records scroll/list/overlay/layout semantics, that `flutter-uiux-to-architecture` has already produced a concrete display-layer decision table and `display_restoration_blueprint`, that important page regions are classified as `preserve_faithfully`, `flutterize`, or `simplify`, that required exported image assets have local paths and intended regions/components, and that the non-display behavior contracts needed by this screen are already stable enough for faithful restoration.
86. Treat hero regions, CTA clusters, branded illustration zones, layered backgrounds, sticky headers, and overlay-heavy regions as fidelity-critical by default unless the frozen prototype-derived design source explicitly says otherwise.
87. During display-layer implementation, keep frozen design-quality guidance active as a guardrail for hierarchy, spacing, typography, contrast, CTA salience, motion restraint, and anti-template composition. These guardrails must not override frozen design-source intent. Use the frozen `display_restoration_blueprint` as the primary code-level contract for page structure, and if corresponding page images exist, inspect them together with the frozen design evidence before landing display-layer code.
88. Treat effect images as visual-structure evidence, not as the only source of truth for Flutter implementation choices. Treat the HTML interactive prototype output as structured restoration evidence, not as independent design guidance. Final scroll, list, sticky, overlay, layout, asset-binding, and region-ownership decisions must follow the frozen global design, the active module design package, `DESIGN.md`, `impl.md`, `flutter-uiux-to-architecture`, and the frozen `display_restoration_blueprint`.
89. If the architecture output or `display_restoration_blueprint` marks a region `preserve_faithfully`, do not normalize away its locked spacing, hierarchy, layering, interaction ownership, or asset usage for engineering convenience. If non-display implementation drifts against that frozen contract, fix the code or return to `flutter-design-source-control` instead of silently rewriting the display layer.
89.1 Do not add page-level manual bottom padding during display-layer restoration unless the frozen `display_restoration_blueprint` explicitly defines an approved bottom content inset. Bottom spacing must come only from frozen ownership such as `SafeArea`, frozen `bottom action area`, or frozen `sticky footer`.
90. If the page effect image or selected design-source packet contains a bitmap visual, texture, illustration, photo, composite, or other image asset that can be downloaded from the approved source, download it and let implementation consume the local asset directly. If the source image is missing, unusable, or explicitly needs generation or rework, let the active MCP / design tool chain choose the concrete generation mechanism.
91. Only generate internal raster visuals when they are genuinely better as image assets. Do not rasterize shapes, simple gradients, icons that belong to an existing vector system, or effects that Flutter can reproduce cleanly with native code.
92. If code is complete or screenshots exist, and both a source visual target plus implementation screenshots are available, first route to `@product-design` `design-qa` as a blocking helper. After that helper passes or returns a review artifact, stop and request explicit human visual inspection against the frozen prototype-derived design source and any approved page effect images. Do not create a separate automatic review workflow stage.
93. If the user requests UI, layout, interaction, hierarchy, visual token, or state changes after shared freeze or module design freeze, use `flutter-design-source-control`.
