# Routing Rules

Use this reference before selecting any downstream skill, changing `current_stage`, or applying module maturity changes.

1. Start by ensuring workflow state is initialized for the current run. If runtime persistence is enabled, initialize the runtime artifact before any stage routing.
2. Before deciding or invoking anything downstream, derive one route lock and run the preflight gate. If either step fails, stop and record a blocker instead of letting the downstream skill reinterpret the workflow.
3. If `confirmation_status` is `pending_confirmation`, do not switch to the next process and do not apply queued status changes, unless `--auto` is active. In `--auto` mode, ordinary downstream confirmation gates are auto-confirmed until the implementation boundary or a blocker is reached. If optional representative effect-image generation is already in scope for the active run, that specific confirmation gate must still stop for user feedback.
4. If the user explicitly confirms a pending transition or status update, promote the queued values, clear pending fields, and continue normal routing from the newly confirmed state.
5. If the user rejects a pending transition or status update, keep the current confirmed state, write the rejection reason, and route back to the skill that must revise artifacts.
6. If a specialist skill returns `blocked`, `not_executed`, or a receipt that fails route-lock validation, do not advance. Keep `current_stage` on the last confirmed stage, clear queued transitions, and record the blocker or route-drift reason.
7. If the input is raw requirements, a one-line feature idea, a partially specified request, or any request without a PRD artifact, set or keep `current_stage=requirements_brainstorming`, read `requirements-prd-flow.md`, brainstorm the requirement space, resolve all decision-blocking questions, and generate a PRD before advancing.
8. If brainstorming finds unresolved decision-blocking questions that cannot be answered from context or by low-risk defaults, do not call `flutter-prd-rd-writer`. Record the question ledger and stop with `required_inputs`.
9. If all PRD-blocking questions are resolved or explicitly defaulted, invoke `flutter-prd-rd-writer` to create or update the PRD artifact, then queue `pending_next_stage=prd_ready` until confirmation unless `--auto` is active.
10. If the input is already a PRD, broad RD, or feature brief, use `flutter-prd-rd-writer` first to derive or refresh the global technical baseline, then queue or promote `technical_baseline_ready`.
11. If a global technical baseline exists but detailed modules and executable module `impl.md` paths do not exist yet, use `flutter-rd-module-splitter` first. Downstream shared visual design, structured design-source work, and module freeze must stay grounded in those executable module contracts.
12. If executable module `impl.md` documents exist but the workflow lacks a brainstormed global visual direction, route to `flutter-taste-router` first. Global visual design must be brainstormed before final product direction confirmation.
13. If the shared visual direction has been brainstormed but the common public shell is not yet explicitly agreed with the user, set or keep `current_stage=shared_taste_direction`, record `required_inputs=public_shell_confirmation`, and stop.
14. If the public shell is agreed but the workflow has not yet chosen a design confirmation path, set or keep `current_stage=shared_taste_direction`, record `required_inputs=design_confirmation_mode:direct_or_subagent_recommendation`, and stop.
15. If `design_confirmation_mode=direct` and the final product design direction has not yet been explicitly confirmed with the user, keep `current_stage=shared_taste_direction`, record `required_inputs=product_design_final_direction_confirmation`, and stop.
16. If `design_confirmation_mode=subagent_recommendation` and no recommendation artifact exists yet, keep `current_stage=shared_taste_direction`, route to a subagent-owned recommendation pass, and queue `pending_next_stage=design_recommendation_ready` on success.
17. If a subagent recommendation artifact exists but the user has not yet confirmed a final direction, set or keep `current_stage=design_recommendation_ready`, record `required_inputs=product_design_final_direction_confirmation`, and stop.
18. If the final product design direction is confirmed after the brainstorming step or after reviewing the recommendation packet, record `current_stage=product_direction_confirmed` or queue that stage according to the confirmation gate.
19. After `product_direction_confirmed`, read `design-md-template.md` and route to output a root-level `DESIGN.md` before entering structured design-source generation, shared freeze, module design freeze, architecture, or code.
20. When `DESIGN.md` is successfully written at the project root, queue or promote `current_stage=design_md_ready`.
21. Do not enter structured design-source generation, shared freeze, module design freeze, architecture, or code before the global visual design direction has been brainstormed, the common public shell has been agreed, the final product direction has been confirmed, and `DESIGN.md` exists.
21. Effect-image generation is optional. Only route into effect-image generation when the user explicitly requests it, when the selected structured design-source adapter needs extra raster evidence, or when `flutter-design-freeze-gate` returns a revision asking for additional visual proof.
22. When optional effect-image generation is in scope, generate exactly one representative light-mode effect image first. Choose the page that best represents the product's overall hierarchy, component family, density, CTA posture, and public-shell language unless the user explicitly names another page.
23. Before representative effect-image generation, check `IMAGE_BASE_URL` and `IMAGE_API_KEY`. If either is missing, stop and record that the optional effect-image branch is blocked by missing image generation credentials.
24. The representative effect image must explicitly include the confirmed product design direction, `DESIGN.md`, PRD constraints, public-shell decisions, art direction, taste constraints, visual system, CTA posture, palette direction, typography mood, component family cues, image-treatment posture, and target page purpose when those values exist.
25. Save the representative effect image into the global effect-image directory under `docs/rd/` using the corresponding page name, index its path in workflow state, set the workflow to wait for user confirmation or revision feedback, and do not generate the remaining optional page effect images yet.
26. After the representative effect image is confirmed, generate the remaining optional all-page light-mode effect-image set only when that broader set is actually needed or explicitly requested.
27. Save every remaining generated page effect image into the same global effect-image directory under `docs/rd/` using the corresponding page name, and index each path in workflow state.
28. All newly generated shared or module-stage effect images must be light-mode visuals. Do not generate dark-mode freeze evidence unless the user explicitly requests a dark-only or dual-mode design cycle.
29. Before any global design freeze or module design freeze, always let `flutter-taste-router` normalize the textual design packet first.
30. Before structured design-source work, require a root-level `DESIGN.md` and a chosen `design_source_adapter`. If the adapter choice is missing or ambiguous, stop with `required_inputs=design_source_adapter:stitch_or_pencil`.
31. If `design_source_adapter=stitch`, read `stitch-design-source.md`, confirm whether the user wants a new Stitch project or an existing Stitch project, then run Stitch page design in page-scoped subagents with at most 6 page-design subagents in parallel, allow those subagents to download needed image assets for direct use, merge their receipts into the structured design-source packet, and preserve any optional effect images as supplemental visual evidence for human visual inspection.
32. If `stitch_project_mode` is missing or ambiguous, stop with `required_inputs=stitch_project_mode:new_or_existing`; do not call Stitch and do not freeze design source. If `stitch_project_mode=existing` and `stitch_project_id` is missing, ambiguous, or not frozen, stop with `required_inputs=stitch_project_id`; do not call Stitch and do not freeze design source. If `stitch_project_mode=new` but project creation cannot return a frozen id, stop with `required_inputs=stitch_project_creation`.
33. If `design_source_adapter=pencil`, read `pencil-design-source.md`, require a frozen Pencil source reference, then run Pencil page design or validation and merge receipts into the structured design-source packet.
34. If the selected structured design-source adapter is unavailable in the current run, record the adapter-specific blocker, keep the relevant freeze blocked, and do not silently fall back to raw effect images as the only design source.
35. If the confirmed `DESIGN.md`, the selected structured design-source output, and any approved optional visual evidence must become a reusable shared design-source contract with fixed global guidance, use `design-preview-to-global-guidelines`, then record `global_guidelines_frozen` as `pending_next_stage` until the user confirms.
36. Before `module_impl_docs_ready`, do not advance into shared visual direction, shared freeze, or module design freeze. First create executable module `impl.md` documents from the technical baseline.
37. If shared design has a candidate direction, confirmed `DESIGN.md`, and a selected structured design-source packet but no explicit freeze decision, use `flutter-design-freeze-gate` directly. Before freezing, explicitly verify that the selected design-source output still preserves the approved style direction, theme system, public shell contract, and shared public component families.
38. If `flutter-design-freeze-gate` finds hierarchy, task guidance, typography, contrast, CTA clarity, or state coverage still weak, do not route to freeze. Allow one shared revision pass through `flutter-taste-router` plus optional effect-image regeneration when needed, then stop unless the user explicitly restarts the design cycle.
39. If the workflow is at or beyond `technical_baseline_ready` but detailed modules and executable module `impl.md` paths do not exist yet, use `flutter-rd-module-splitter` to create module index rows and implementation-final `impl.md` documents in one pass.
40. If multiple feature modules depend on one shared route host, root redirect layer, tab shell, or shell-level state, allow `flutter-rd-module-splitter` to split an `app-shell` or `root-shell` module first instead of burying that responsibility inside feature modules.
41. If a selected module lacks an executable `docs/rd/modules/<module>/<module>.impl.md`, route back to `flutter-rd-module-splitter` for a scope-matched regeneration instead of opening a separate refinement stage.
42. Module implementation preparation consumes the executable `impl.md` produced by `flutter-rd-module-splitter`. Do not require or create `docs/rd/modules/<module>/<module>.ui-ux.md`.
43. When `flutter-rd-module-splitter` produces executable module `impl.md` documents, keep `current_stage` at the last confirmed stage, queue `pending_next_stage=module_impl_docs_ready`, queue each generated module's `impl_status=implementation_final`, and wait for confirmation before shared visual design. In `--auto` mode, auto-apply those queued updates and continue directly into shared visual direction.
44. In module freeze, the frozen selected structured design-source packet is required. If the active module lacks that packet, stop and route back to the selected design-source generation or validation path.
45. If a complete active-module visual draft, optional effect-image pack, or implementation-facing design-source packet exists, route to the selected design-source generation or validation path first, then route to `flutter-design-freeze-gate` before queueing `module_design_frozen`. Treat the module's high-fidelity visual contract as the first freeze question.
46. During executable module document generation and module freeze preparation, do not generate new module real-device effect images by default. Generate them only when `--perviewer` is explicitly active for the current run.
47. When `--perviewer` is active and module-stage effect images are generated, save them into the global effect-image directory under `docs/rd/`, then write that opt-in plus the generated global effect-image paths into `global-design-guidelines.md`.
48. If `flutter-design-freeze-gate` finds the active-module design package not yet strong enough to freeze, keep the workflow on the active module, update the selected design-source packet, module `impl.md`, and optional visual evidence exactly once, and stop without another automatic freeze pass unless the user explicitly restarts a design cycle.
49. When `flutter-design-freeze-gate` approves the active module design source, first verify its receipt includes `high_fidelity_freeze_status=passed` or an explicit approved reduction. Do not immediately mark it frozen. Queue `pending_next_stage=module_design_frozen`, queue `design_source_status=frozen`, and if `impl.md` references the frozen selected structured design-source packet, also queue `impl_status=landed`.
50. If a frozen selected structured design source is about to be consumed by implementation RD or code, use `flutter-design-source-control`.
51. If frozen structured design-source output, theme values, component rules, and optional visual evidence must become Flutter-facing tokens, assets, components, screen architecture, and non-native visual fallback decisions, route to `flutter-uiux-to-architecture`.
52. Before architecture planning, implementation-readiness confirmation, human visual inspection, or code implementation, verify that `platform_identifier` is explicit enough for the real validation surface. Do not treat `platform_baseline` such as `ios_hig` as if it already meant `android_emulator`, `windows_desktop`, or `web_browser`.
53. Do not move a module into `implementing` until `technical_baseline_ready`, `module_impl_docs_ready`, `module_design_frozen`, `impl_rd_ready`, and `bootstrap_code_ready` exist for the module, confirmed maturity is at least `impl_status=landed` and `design_source_status=frozen`, the frozen selected structured design-source packet exists, and the required global public code baseline is already landed.
54. The required global public code baseline before module implementation must include app bootstrap and environment initialization, root router or route host plus root redirect policy, global dependency injection or provider scope entry, local storage baseline and persistence wiring, global error mapping and logging baseline, the shared theme or design-token baseline required by the frozen selected structured design source, and the shared shell layer when feature modules depend on an `app-shell` or `root-shell`.
55. Add network baseline and API client wiring to the global public code baseline only when the project or target modules actually require remote data, API access, upload, sync, or other network capabilities.
56. As soon as the shared public baseline is explicit enough, prefer triggering `flutter-init` before feature-module code begins.
57. Do not wait for every feature module to reach `architecture_ready` before triggering `flutter-init` when the shared baseline is already clear enough.
58. If the shared bootstrap-critical baseline is ready and the target project has not been scaffolded yet or does not contain project-local `skills/flutter-dev/`, use `flutter-init` before feature-module implementation begins.
59. `flutter-init` must stop at initialization boundaries. It may create the directory skeleton and project-local `skills/flutter-dev/` container, but it must not implement bootstrap code, shared wiring, feature code, page code, or module-specific business logic.
60. If `flutter-init` has completed and the directory skeleton plus project-local `skills/flutter-dev/` exist, record `project_initialized` as `pending_next_stage`.
61. After `project_initialized` is confirmed, route to the separate bootstrap code stage.
62. `project_initialized` is directory-only. Do not treat it as proof that bootstrap code, shared wiring, or any feature implementation already exists.
63. If bootstrap code has landed and the required global public baseline is actually present on disk, record or queue `bootstrap_code_ready`.
64. If `--auto` is active, do not stop for `project_initialized`, `bootstrap_code_ready`, `implementation_final`, `module_design_frozen`, `impl_rd_ready`, `architecture_ready`, or other downstream confirmation gates. Keep advancing until the implementation boundary is reached for all target modules or a blocker appears.
65. If `--auto` is active and all target modules are implementation-ready, stop here instead of entering `implementing`.
66. If implementation work should begin, first route through `@superpowers` to produce `Spec` for the active module. Do not begin execution before that `Spec` exists.
67. After `Spec` is confirmed, route through `@superpowers` again to produce `Plan` for the active module. Do not begin code execution before that `Plan` exists.
68. Only after both `Spec` and `Plan` exist may implementation execution begin through explicit `@superpowers` together with project-local `flutter-dev` and `flutter-project-guardrails`.
69. Implementation execution defaults to parallel execution of independent work packages defined by `Plan`. The orchestrator should split the active module into non-conflicting ownership units and dispatch them in parallel unless shared-file or shared-state conflicts make serial execution necessary.
70. During module implementation, at minimum consider splitting execution into design-source display fidelity and implementation behavior tracks when the work can be owned independently.
71. The default module landing order is: define the minimum data contract first, then land the display layer skeleton and main user path from the frozen selected structured design source, then connect the real data layer.
72. For ordinary business-page modules, prefer `minimum data contract -> display layer -> data integration`. For infrastructure-heavy modules, it is allowed to land the minimum process or data layer first and then attach display surfaces later.
73. Before display-layer code begins, run a display-layer readiness preflight. Verify that the page has enough readable visual evidence when needed, that the selected structured design source records scroll/list/overlay/layout semantics, that `flutter-uiux-to-architecture` has already produced a concrete display-layer decision table, that important page regions are classified as `preserve_faithfully`, `flutterize`, or `simplify`, and that required downloaded image assets have local paths and intended regions/components.
74. Treat hero regions, CTA clusters, branded illustration zones, layered backgrounds, sticky headers, and overlay-heavy regions as fidelity-critical by default unless the frozen selected structured design source explicitly says otherwise.
75. During display-layer implementation, keep taste guidance active as a guardrail for hierarchy, spacing, typography, contrast, CTA salience, motion restraint, and anti-template composition. Taste must not override frozen design-source intent. If corresponding page images exist, inspect them through `$image-to-code` before landing display-layer code.
76. Treat effect images as visual-structure evidence, not as the only source of truth for Flutter implementation choices. Final scroll, list, sticky, overlay, and layout decisions must follow the combination of the frozen selected structured design source, `impl.md`, and `flutter-uiux-to-architecture`.
77. If the architecture output marks a region `preserve_faithfully`, do not normalize away its locked spacing, hierarchy, layering, or asset usage for engineering convenience. Any such change must return to `flutter-design-source-control`.
78. If the page effect image or selected design-source packet contains a bitmap visual, texture, illustration, photo, composite, or other image asset that can be downloaded from the approved source, download it and let implementation consume the local asset directly. Use `$imagegen` only when the source image is missing, unusable, or explicitly needs generation or rework.
79. Only use `$imagegen` for visuals that are genuinely better as raster assets. Do not use it for shapes, simple gradients, icons that belong to an existing vector system, or effects that Flutter can reproduce cleanly with native code.
80. If code is complete or screenshots exist, stop and request human visual inspection against the frozen selected structured design source and any approved page effect images. Do not create a separate automatic review workflow stage and do not route to an automatic visual reviewer by default.
81. If the user requests UI, layout, interaction, hierarchy, visual token, or state changes after shared freeze or module design freeze, use `flutter-design-source-control`.
