# Routing Rules

Use this reference before selecting any downstream skill, changing `current_stage`, or applying module maturity changes.

1. Start by ensuring workflow state is initialized for the current run. If runtime persistence is enabled, initialize the runtime artifact before any stage routing.
2. Before deciding or invoking anything downstream, derive one route lock and run the preflight gate. If either step fails, stop and record a blocker instead of letting the downstream skill reinterpret the workflow.
3. If `confirmation_status` is `pending_confirmation`, do not switch to the next process and do not apply queued status changes, unless `--auto` or `--full-auto` is active. In `--auto` mode, ordinary downstream confirmation gates are auto-confirmed until workflow completion or a blocker is reached. In `--full-auto`, deterministic human-facing gates may also be auto-confirmed when the control contracts prove there is exactly one supported default.
4. If the user explicitly confirms a pending transition or status update, promote the queued values, clear pending fields, and continue normal routing from the newly confirmed state.
5. If the user rejects a pending transition or status update, keep the current confirmed state, write the rejection reason, and route back to the skill that must revise artifacts.
6. If a specialist skill returns `blocked`, `not_executed`, or a receipt that fails route-lock validation, do not advance. Keep `current_stage` on the last confirmed stage, clear queued transitions, and record the blocker or route-drift reason.
7. If the input is still a rough idea, reference bundle, or abstract request that is not yet concrete enough to discuss through pages, set or keep `current_stage=idea_sketch_brainstorming`, read `idea-sketch-flow.md`, brainstorm the product direction, and generate `1-3` key-page sketches before any PRD work begins.
8. If `idea_sketch_brainstorming` has not yet produced confirmed key-page sketches, do not call `requirements-prd-flow.md`, `flutter-prd-rd-writer`, technical baseline generation, Product Design brief confirmation, shared freeze, architecture, or implementation planning. Record `required_inputs=idea_sketch_confirmation` and stop.
9. Before leaving `idea_sketch_brainstorming`, ensure the user has confirmed product direction, page structure, and broad visual style, and ensure `docs/project/idea-sketch-brief.md` exists on disk.
10. If the idea-sketch confirmation has passed and `docs/project/idea-sketch-brief.md` exists, queue `pending_next_stage=requirements_brainstorming` until confirmation unless `--auto` or `--full-auto` is active.
11. If the input is already concrete enough for PRD work but no PRD artifact exists yet, or if the workflow has just confirmed `docs/project/idea-sketch-brief.md`, set or keep `current_stage=requirements_brainstorming`, read `requirements-prd-flow.md`, brainstorm the requirement space, resolve all decision-blocking questions, and generate a PRD before advancing.
12. If requirements brainstorming finds unresolved decision-blocking questions that cannot be answered from context or by low-risk defaults, do not call `flutter-prd-rd-writer`. Record the question ledger and stop with `required_inputs`.
13. Before promoting the run to `prd_ready`, read `prd-template.md` and `prd-completeness-gate.md`, ensure the PRD artifact exists on disk, ensure the question ledger is embedded or linked, and ensure the completeness outcome is `ready_for_prd_ready`.
14. If all PRD-blocking questions are resolved or explicitly defaulted and the completeness gate passes, queue `pending_next_stage=prd_ready` until confirmation unless `--auto` or `--full-auto` is active.
15. If the input is already a PRD, broad RD, or feature brief, use `flutter-prd-rd-writer` first to derive or refresh the global technical baseline, then queue or promote `technical_baseline_ready`.
16. If a global technical baseline exists but the target design-device preset and base resolution are not yet frozen for this design cycle, keep `current_stage=technical_baseline_ready`, record `required_inputs=design_device_preset_and_resolution`, and stop. In manual mode, offer at least these common iPhone presets unless the user already chose another viewport: `375 x 667 px`, `390 x 844 px`, `393 x 852 px`, and `430 x 932 px`. In `--auto` or `--full-auto`, if no frozen viewport exists yet, auto-freeze `390 x 844 px` as the design viewport and continue.
17. If a global technical baseline exists but the workflow lacks a confirmed Product Design brief for this request, first read `design-quality-guidance.md`, then route to `@product-design` `get-context`. Product design must be briefed before final product direction confirmation.
18. If a global technical baseline exists but the workflow lacks a confirmed structured Product Design clarification packet for later module splitting, keep the run on the shared-direction branch and route to `@product-design` scope clarification before any shared freeze completion or module splitting. That clarification must lock core user journeys, page families, critical states, interaction goals, explicit `platform_identifier`, platform-aware shell or navigation expectations, and per-surface information-density posture.
19. Do not generate module `impl.md` documents, module implementation-stage effect images, or module freeze artifacts before the shared/global design freeze has completed.
20. If the common public shell is not yet explicitly agreed with the user, set or keep `current_stage=shared_design_direction`, record `required_inputs=public_shell_confirmation`, and stop. In `--full-auto`, if the current shell packet has exactly one supported shared-shell contract and no competing shell option remains in scope, auto-confirm it and continue instead of stopping.
21. If the Product Design brief for this request has not yet been confirmed, set or keep `current_stage=shared_design_direction`, route to `@product-design` `get-context`, and stop until the brief is confirmed.
22. If the Product Design brief is confirmed but the workflow has not yet completed a Mobbin-backed inspiration pass for the current design cycle, keep `current_stage=shared_design_direction`, route to `@product-design` through a Mobbin-first shared/global style recommendation pass, and stop any direction recommendation, representative-sketch generation, final effect-image generation, or downstream freeze work until that pass is complete.
23. If the Product Design brief is confirmed and the Mobbin-backed inspiration pass is complete but no representative sketch artifact exists yet, keep `current_stage=shared_design_direction`, route to `@product-design` visual ideation using the approved reference images plus a Mobbin-backed shared/global style recommendation pass, then generate the representative local sketch through `$imagegen` under the confirmed Product Design brief and commercial product UI/UX constraints, run one automatic `@product-design` QA pass on that generated sketch, and queue `pending_next_stage=design_recommendation_ready` on success. Treat `https://mobbin.com/` only as inspiration evidence for this pass, not as the formal downstream design source.
24. If a representative sketch recommendation artifact exists but the user has not yet confirmed a final direction, set or keep `current_stage=design_recommendation_ready`, record `required_inputs=product_design_final_direction_confirmation`, and stop. In manual mode, this confirmation step should expose exactly three recommended representative directions with one primary recommendation plus the generated representative sketch instead of a single hidden answer. In `--full-auto`, if one clear primary recommendation exists and no competing approved direction remains unresolved, auto-confirm that direction and continue.
25. If the workflow needs richer campaign, hero, mood-board, or commercial visual evidence before the representative sketch direction can be chosen confidently, keep the run on the shared-direction branch, read `creative-production-branch.md`, route into the pre-direction Creative Production branch, and return to Product Design confirmation with that evidence.
26. If the user already supplied a concrete approved visual source and confirms the generated representative sketch as the final direction after Product Design brief playback, use that approval to satisfy the recommendation step, then call `gpt-image-2-generator` for the representative final effect image, and record `current_stage=product_direction_confirmed` or queue that stage according to the confirmation gate only after that final effect image exists.
27. If the final product design direction is confirmed after the representative-sketch recommendation pass, route to `effect-image-to-ui-sheet-atlas` instead of directly calling `gpt-image-2-generator`. That node must generate the representative final effect image, the matching UI-only atlas, the atlas manifest, and the atlas slicing config under the same confirmed Product Design brief, accepted direction, and commercial product UI/UX constraints, then run the required automatic `@product-design` QA pass on the generated image before queueing `pending_next_stage=ui_sheet_atlas_ready`. In `--auto`, if a strong primary recommended style exists among the three compared Mobbin-backed directions and no blocker remains, adopt that recommended style directly, generate the representative sketch, and continue into atlas preparation without pausing for manual confirmation. In `--full-auto`, treat that same strong primary recommendation as sufficient final-direction confirmation when no competing approved direction remains unresolved, but still generate the representative sketch before atlas preparation.
27.1. If the current scope is at `ui_sheet_atlas_ready`, do not advance to `DESIGN.md` yet. First require atlas-bundle confirmation in manual mode, then route to `ui-sheet-atlas-slicer`.
27.2. When `ui-sheet-atlas-slicer` succeeds for the shared representative page, queue `pending_next_stage=product_direction_confirmed` and record the slice manifest before continuing.
28. After `product_direction_confirmed`, read both `design-quality-guidance.md` and `design-md-template.md`, then route to output a root-level `DESIGN.md` before entering any later structured design-source normalization, shared freeze, module design freeze, architecture, or code.
29. When `DESIGN.md` is successfully written at the project root, queue or promote `current_stage=design_md_ready`.
29.1. Route to `effect-image-to-pencil-design` first. The shared Pencil design source is mandatory and must be compared against the approved effect image, reviewed through a visible-difference list plus concrete repair actions, repaired once after confirmation, and accepted by a human before any new shared bitmap-generation scope is finalized.
29.1.1. During shared Pencil design execution, keep the frozen design width unchanged and do not let any Pencil page height fall below the frozen base design viewport height. If a page or major region is intentionally scrollable, allow the Pencil page height to exceed that minimum so the designed scroll range remains visible.
29.2. Only after `pencil_design_source_ready` is confirmed, if the shared scope still contains page regions whose confirmed post-review design cannot be restored faithfully enough in code and therefore still needs bitmap assets, read `asset-atlas-flow.md` plus `global-asset-catalog-contract.md`, update or repair `docs/project/assets/global-asset-catalog.json`, and route through shared per-resource generation.
29.3. During shared resource generation, work on one shared page at a time. First exclude any image region that is only a schematic placeholder whose real content must be created later from runtime data. Record that region as a placeholder contract, not as a generated asset. For the remaining bitmap assets on that page, first check `docs/project/assets/global-asset-catalog.json` plus the approved output paths it references. If an approved reusable image already exists for the same semantic and usage scenario, reuse it directly and remove it from new generation scope. For the still-unresolved bitmap assets, classify every detected asset as `reusable`, `candidate_reuse`, or `shared_only` by `name`, `semantic`, and `usage_scenarios` rather than by filename alone. If the catalog cannot be loaded or validated, stop with blocker `missing_global_asset_catalog`.
29.4. If any shared-scope asset remains `candidate_reuse`, record `required_inputs=asset_reuse_confirmation`, stop with blocker `candidate_reuse_unresolved`, and do not continue into resource generation or shared freeze preparation.
29.5. After reuse filtering for the current shared page, record `required_inputs=page_bitmap_list_confirmation` and stop until the user confirms that page's post-review new bitmap list. Do not generate image assets before this confirmation.
29.6. Queue or promote `current_stage=shared_asset_resources_ready` only after the shared flow has generated each approved resource independently through `$imagegen`, run one automatic `@product-design` QA pass on each generated shared image asset, saved the final asset files, and recorded their output paths. If any required asset file is missing, stop with blocker `missing_generated_asset`.
29.7. If shared generated assets exist but are not yet approved, record `required_inputs=asset_generation_confirmation` and stop before shared freeze preparation.
29.8. Only after Pencil review is confirmed, and after any still-required shared asset generation is complete, may the orchestrator continue into shared freeze preparation.
30. Do not enter shared Pencil design generation, shared freeze, module design freeze, architecture, or code before the global visual design direction has been brainstormed, the common public shell has been agreed, the final product direction has been confirmed, and `DESIGN.md` exists.
31. Before shared Pencil design execution or shared freeze preparation, verify that `DESIGN.md` explicitly covers at least task hierarchy, first-screen CTA posture, interaction and feedback rules, responsive strategy, critical states and edge cases, content tone when those concerns are relevant to the product, the frozen base design viewport or device preset, and the platform-aware information-density posture for major surfaces. If that coverage is weak, route back to revise `DESIGN.md` before continuing.
31.1. If the shared scope still needs non-standard bitmap assets after Pencil review and `shared_asset_resources_ready` is still absent, record `required_inputs=missing_generated_assets` and stop. Shared structured design sources must reference final approved generated asset files for every approved bitmap asset, not screenshot crops or undeclared temporary placeholder assets. Regions that are explicitly tracked as `placeholder_only` may remain placeholders.
32. If the request is primarily about campaign visuals, ads, mood boards, launch assets, social cards, hero directions, scenes, shots, logos, or other asset-oriented creative work, read `creative-production-branch.md` and route into `Creative Production:explore` once the brief is specific enough.
33. Before final product direction confirmation, treat the Creative Production branch only as direction input. After `design_md_ready`, treat it as the post-direction asset-production branch.
34. If the user already selected an asset family or asks directly for one of the focused Creative Production paths, skip the generic chooser and route directly to the scope-matched explorer: `moodboard-explorer`, `ads-explorer`, `offer-explorer`, `scene-explorer`, `shot-explorer`, or `logo-explorer`.
35. If the user needs publish-bound marketing finish, exact-size social outputs, or premium visual polish while preserving exact text, logos, charts, data, safe zones, and dimensions, route to `Creative Production:generative-polish` only after one direction or deterministic base is already selected.
36. Do not use the Creative Production branch to replace Product Design brief confirmation, product-surface direction selection, `DESIGN.md`, or shared freeze. Treat it as an asset branch, not as a shortcut around the app-design workflow.
37. Do not route broad marketing or campaign exploration straight to `gpt-image-2-generator` when the Creative Production branch is available and the user needs reviewable creative routes, boards, or publish-safe asset production. Use representative-sketch generation first for the app-preview branch, and use final effect-image generation only after the sketch is confirmed or when the user explicitly wants lightweight page evidence instead of the richer asset workflow.
38. If a Creative Production run surfaces a materially different visual direction from the already confirmed product direction, stop downstream freeze or implementation promotion, route back to Product Design confirmation, and revise `DESIGN.md` before continuing.
39. Final effect-image generation is mandatory in both manual mode and `--auto` / `--full-auto`, but the shared/global route must enter representative sketch generation first. Route into representative sketch generation immediately after the Product Design brief is confirmed, using the approved reference images and Product Design direction constraints as the source, then call `gpt-image-2-generator` only after that sketch is confirmed or auto-adopted by the active execution mode.
40. In manual mode, reuse the approved Product Design visual target and reference images as the representative light-mode sketch baseline whenever possible. If no approved representative visual target exists yet, stop and route back to the Product Design recommendation branch instead of inventing a new visual baseline.
41. Before representative final effect-image generation, verify that `gpt-image-2-generator` is available and authenticated. If access, credentials, or upstream visual generation capability is missing, stop and record the branch-specific blocker. Representative sketch generation before that gate must use local `$imagegen`.
42. Before any shared/global or module-stage representative-sketch or final effect-image generation request is sent, read the frozen global design packet and extract the applicable visual constraints for the target page or module. Generated sketches and final effect images must explicitly inherit the confirmed product design direction, the approved Product Design visual target, the frozen base design viewport, `DESIGN.md`, PRD constraints, public-shell decisions, art direction, visual system, CTA posture, palette direction, typography mood, component family cues, image-treatment posture, target page purpose, and any first-screen task priority that materially affects the page when those values exist. Both stages must remain within `@product-design` design control and must read as mature commercial product UI/UX instead of concept art or marketing collateral. Keep the frozen design width unchanged, treat the frozen viewport height as the minimum image height, and allow taller outputs only when the target page or region is intentionally scrollable. The atlas-preparation node must also produce a matching UI-only transparent atlas, atlas manifest, and atlas slicing config whose background stays transparent and whose runtime-data regions remain excluded from cut-safe slices. If generation fails and a retry is needed, resend the full original prompt plus the same frozen constraints and any blocker-specific repair notes; do not simplify, shorten, or strip prompt content for the retry.
43. In manual mode, save the representative sketch into the global effect-image directory under `docs/project/` using the corresponding page name plus a `-draft` suffix, run one automatic `@product-design` QA pass on that generated sketch, index both the sketch path and QA result in workflow state, write the generated image row into `docs/project/assets/global-asset-catalog.json`, set the workflow to wait for user confirmation or revision feedback, and do not generate the representative final effect image or the remaining required page effect images yet.
44. In manual mode, after the representative sketch is confirmed, generate the representative final effect image through `gpt-image-2-generator`, run one automatic `@product-design` QA pass on that final image, freeze that final effect image as the current-cycle visual baseline, write the generated image row into `docs/project/assets/global-asset-catalog.json`, create the matching UI-only transparent atlas plus atlas manifest, delete previously generated alternative sketches or effect images that are no longer part of the frozen packet, update indexes plus the same catalog rows to keep only the active baseline, and record that alternative-direction browsing is closed. Do not reopen Product Design direction browsing or generate other-direction images inside the same design cycle.
45. In `--auto` or `--full-auto`, once the representative effect-image direction is chosen, run one automatic `@product-design` QA pass on every generated image before freezing it as the current-cycle visual baseline, write the generated image rows plus QA results into `docs/project/assets/global-asset-catalog.json` or the workflow record, and continue without inventing an extra all-page effect-image stage.
46. All newly generated shared or module-stage effect images must be light-mode visuals. Do not generate dark-mode freeze evidence unless the user explicitly requests a dark-only or dual-mode design cycle.
47. Before any global design freeze or module design freeze, keep `@product-design` as the primary design controller. If the frozen packet still needs normalization after the Product Design direction is already confirmed, normalize it from approved images and confirmed design artifacts, then route through mandatory Pencil design execution and any still-required bitmap generation before shared freeze.
48. Before Pencil design execution or later bitmap generation, require a root-level `DESIGN.md`, the frozen global design packet, and the active module design package when module scope is involved.
51. If the confirmed `DESIGN.md`, the selected shared structured design-source packet, and the approved required effect-image evidence must become a reusable shared design-source contract with fixed global guidance, use `design-preview-to-global-guidelines`, then record `global_guidelines_frozen` as `pending_next_stage` until the user confirms.
52. Before `design_freeze_ready` and the shared/global design freeze complete, do not advance into any module-related workflow. Module `impl.md` generation, module implementation-stage effect-image generation, module freeze, module architecture, and module code all stay behind the shared freeze gate.
53. If shared design has a candidate direction, confirmed `DESIGN.md`, and a selected structured design-source packet but no explicit freeze decision, use `flutter-design-freeze-gate` directly. Before freezing, explicitly verify that the selected design-source output still preserves the approved style direction, theme system, public shell contract, shared public component families, task hierarchy, interaction feedback behavior, responsive strategy, and critical-state language.
54. If `flutter-design-freeze-gate` finds hierarchy, task guidance, typography, contrast, CTA clarity, state coverage, feedback consistency, responsive strategy, or content-tone coverage still weak, do not route to freeze. Allow one shared revision pass through the approved image-backed design-source flow, `DESIGN.md` refinement, plus required effect-image regeneration when needed, then stop unless the user explicitly restarts the design cycle.
55. Once the shared/global design freeze is complete and the shared bootstrap-critical baseline is explicit enough, prefer `flutter-init` immediately instead of waiting for later module architecture milestones.
56. If the shared bootstrap-critical baseline is ready and the target project has not been scaffolded yet or the workspace does not contain sibling `skills/flutter-dev/`, use `flutter-init` before feature-module implementation begins.
57. `flutter-init` must stop at initialization boundaries. It may create the directory skeleton and sibling `skills/flutter-dev/` container, but it must not implement bootstrap code, shared wiring, feature code, page code, or module-specific business logic. Unless the user explicitly overrides the environment contract, route initialization and later command guidance under the assumption that Flutter is managed through `fvm`.
58. If `flutter-init` has completed and the directory skeleton plus sibling `skills/flutter-dev/` exist, record `project_initialized` as `pending_next_stage`.
59. After `project_initialized` is confirmed, route to the separate bootstrap code stage.
60. `project_initialized` is directory-only. Do not treat it as proof that bootstrap code, shared wiring, or any feature implementation already exists.
61. If bootstrap code has landed and the required global public baseline is actually present on disk, record or queue `bootstrap_code_ready`.
62. If the shared/global design freeze is complete but detailed modules and executable module `impl.md` paths do not exist yet, first verify that the confirmed Product Design clarification packet for module splitting is present. If it is missing, route back to `@product-design` scope clarification and stop. Otherwise use `flutter-rd-module-splitter` to create module index rows and implementation-final `impl.md` documents in one pass. Those module documents must explicitly inherit the frozen shared design direction, shared interaction principles, shared public shell decisions, shared component constraints, platform-specific interaction expectations, and approved information-density posture.
62.1 Before accepting a module split, verify that each page family has exactly one owning module.
62.2 If two candidate modules still describe the same primary user job, same core data owner, and same state lifecycle, do not approve the split yet. Keep the pages in one module unless a real permission boundary, release slice boundary, or independent backend ownership proves the separation.
63. If multiple feature modules depend on one shared route host, root redirect layer, tab shell, or shell-level state, allow `flutter-rd-module-splitter` to split an `app-shell` or `root-shell` module first instead of burying that responsibility inside feature modules, but still only after the shared/global design freeze is complete.
64. If a selected module lacks an executable `docs/project/modules/<module>/<module>.impl.md`, route back to `flutter-rd-module-splitter` for a scope-matched regeneration instead of opening a separate refinement stage.
65. Module implementation preparation consumes the executable `impl.md` produced by `flutter-rd-module-splitter`. Treat that `impl.md` as a detailed module task implementation document constrained by the frozen shared design and interaction principles. Do not require or create `docs/project/modules/<module>/<module>.ui-ux.md`.
66. When `flutter-rd-module-splitter` produces executable module `impl.md` documents, keep `current_stage` at the last confirmed stage, queue `pending_next_stage=module_impl_docs_ready`, queue each generated module's `impl_status=implementation_final`, and wait for confirmation before the active module's native HTML preparation. In `--auto` or `--full-auto` mode, auto-apply those queued updates, select the first unfinished module in the confirmed serial module order, and continue directly into that active module's native HTML path.
67. After `module_impl_docs_ready`, route the active module to a native HTML prototype step before any module-scoped atlas preparation. That step must render the module pages' static layout with only native `HTML`, `CSS`, and `JavaScript`; do not allow any framework, component runtime, or generated SPA shell. Unless the user explicitly overrides the language, the visible UI copy in that prototype must be Simplified Chinese. The prototype must read like a mature commercial product page, not a marketing surface, ad-style composition, or explanation-heavy mock.
67.1. Before queueing `module_html_prototype_ready`, verify that the active module `impl.md` already fixes the module function, required page-local states, main task path, and non-display behavior boundaries strongly enough for page-level HTML expression. If required states such as `error`, `empty`, `loading`, permission, or other interaction-critical states are still missing, route back to supplement them first.
67.2. In manual mode, once the native HTML prototype is reviewable, keep `current_stage` on the last confirmed stage, queue `pending_next_stage=module_html_prototype_ready`, record the prototype receipt, and stop for explicit user confirmation. Do not start atlas preparation, atlas slicing, display-evidence confirmation, or module Pencil design execution before that confirmation.
68. After the native HTML prototype is confirmed, route the active module to `effect-image-to-ui-sheet-atlas`. That node must generate or refresh the module effect image together with the matching UI-only transparent atlas, atlas manifest, and atlas slicing config under frozen shared constraints, then queue `pending_next_stage=ui_sheet_atlas_ready`. If the page lacks the required main preview, key state scope, or atlas-preparation inputs for fidelity-critical regions, stop and repair the evidence instead of inferring missing structure from the happy-path page alone.
68.1. After the module atlas bundle is confirmed, route to `ui-sheet-atlas-slicer`. When slicing succeeds, queue `pending_next_stage=ui_sheet_atlas_slices_ready` and record the slice manifest.
68.2. After atlas slicing succeeds, confirm `display_evidence_pack_ready` for the active module and then route to `effect-image-to-pencil-design`. The module Pencil design source is mandatory and must be compared against the approved implementation-stage effect image, reviewed through a visible-difference list plus concrete repair actions, repaired once after confirmation, and accepted by a human before any new module bitmap-generation scope is finalized.
68.1.1. During module Pencil design execution, keep the frozen design width unchanged and do not let any Pencil page height fall below the frozen base design viewport height. If a page or major region is intentionally scrollable, allow the Pencil page height to exceed that minimum so the designed scroll range remains visible.
68.2. Only after `pencil_design_source_ready` is confirmed, if the active module still contains page regions whose confirmed post-review design cannot be restored faithfully enough in code and therefore still needs bitmap assets, read `asset-atlas-flow.md` plus `global-asset-catalog-contract.md`, query `docs/project/assets/global-asset-catalog.json` first, and route through module per-resource generation.
68.3. During module resource generation, work on one module page at a time. Exclude any image region that is only a schematic placeholder whose real content must be created later from runtime data. Record that region as a placeholder contract, not as a generated asset. For the remaining bitmap assets on that page, first check `docs/project/assets/global-asset-catalog.json` plus the approved output paths it references. If an approved reusable image already exists for the same semantic and usage scenario, reuse it directly and remove it from new generation scope. For the still-unresolved bitmap assets, mark semantically close but unresolved reuse cases as `candidate_reuse`, record `required_inputs=asset_reuse_confirmation`, and stop with blocker `candidate_reuse_unresolved`. Do not silently merge or regenerate them.
68.4. After reuse filtering for the current module page, record `required_inputs=page_bitmap_list_confirmation` and stop until the user confirms that page's post-review new bitmap list. Do not generate image assets before this confirmation.
68.5. Queue or promote `current_stage=module_asset_resources_ready` only after the module flow has updated the global catalog, generated each approved resource independently through `$imagegen`, run one automatic `@product-design` QA pass on each generated image asset, and saved the final asset files into the page asset directory. If any required asset file is missing, stop with blocker `missing_generated_asset`.
68.6. If module generated assets exist but are not yet approved, record `required_inputs=asset_generation_confirmation` and stop before `pencil_restoration_ready`.
68.7. Only after Pencil review is confirmed, and after any still-required module asset generation is complete, may the orchestrator continue into `pencil_restoration_ready`.
68.8. Module structured design sources must directly reference shared or module generated asset files for every approved bitmap asset rather than screenshot crops or undeclared temporary placeholder assets. Regions that are explicitly tracked as `placeholder_only` may remain placeholders.
69. Do not generate module real-device effect images or module bitmap assets during executable module document generation. First finish the native HTML confirmation gate, then finish atlas preparation and atlas slicing inside the implementation-stage visual restoration loop.
70. When module-stage atlas bundles are generated, save the effect images into the global effect-image directory under `docs/project/`, save the matching UI-only transparent atlas plus atlas manifest plus atlas slicing config into the atlas artifact tree, run one automatic `@product-design` QA pass on each generated image, and then write the generated effect-image paths into `global-design-guidelines.md` and the matching generated-image rows into `docs/project/assets/global-asset-catalog.json`. When the slicing node succeeds, save the slice result manifest beside the atlas bundle and index it in the workflow record.
71. Before queueing `module_design_frozen`, verify that the active module already has an accepted native HTML prototype, the selected implementation-stage effect image, the matching atlas evidence, the atlas slice evidence, the display evidence pack, and the mandatory module Pencil design source. If any of those artifacts are missing or weak, keep the workflow on the active module, update the selected design-source packet, module `impl.md`, native HTML prototype, atlas bundle, slice bundle, or other approved visual evidence exactly once, and stop without another automatic freeze pass unless the user explicitly restarts a design cycle.
72. When `flutter-design-freeze-gate` approves the active module design source, first verify its receipt includes `high_fidelity_freeze_status=passed` or an explicit approved reduction. Before marking it frozen, delete obsolete pre-freeze module draft evidence that is no longer selected for this frozen packet and clear its workflow references. Do not immediately mark it frozen. Queue `pending_next_stage=module_design_frozen`, queue `design_source_status=frozen`, and if `impl.md` references the frozen structured design-source packet, also queue `impl_status=landed`.
73. If a frozen structured design source is about to be consumed by implementation RD or code, use `flutter-design-source-control`.
74. If a frozen Pencil design source must first become a Flutter restoration contract, route to `pencil-design-to-flutter-restoration`. After that contract exists, route to `flutter-uiux-to-architecture` so the frozen structured design-source output, restoration decisions, theme values, component rules, required effect-image evidence, and any additional approved visual evidence become Flutter-facing tokens, assets, components, screen architecture, display-layer decision tables, and non-native visual fallback decisions.
75. Before architecture planning, implementation-readiness confirmation, human visual inspection, or code implementation, verify that `platform_identifier` is explicit enough for the primary runtime and validation platform. Do not treat `platform_baseline` such as `ios_hig` as if it already meant a runnable primary platform. Module-level design must already have considered this concrete primary platform before architecture inherits the packet.
