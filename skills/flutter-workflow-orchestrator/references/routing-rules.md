# Routing Rules

Use this reference before selecting any downstream skill, changing `current_stage`, or applying module maturity changes.

1. Start by ensuring `docs/rd/00-workflow-record.md` exists. If it is missing, create it before any stage routing.
2. Before deciding or invoking anything downstream, derive one route lock and run the preflight gate. If either step fails, stop and record a blocker instead of letting the downstream skill reinterpret the workflow.
3. If `confirmation_status` is `pending_confirmation`, do not switch to the next process and do not apply queued status changes, unless `--auto` is active. In `--auto` mode, ordinary downstream confirmation gates are auto-confirmed until the implementation boundary or a blocker is reached, but the representative effect-image confirmation gate must still stop for user feedback.
4. If the user explicitly confirms a pending transition or status update, promote the queued values, clear pending fields, and continue normal routing from the newly confirmed state.
5. If the user rejects a pending transition or status update, keep the current confirmed state, write the rejection reason, and route back to the skill that must revise artifacts.
6. If a specialist skill returns `blocked`, `not_executed`, or a receipt that fails route-lock validation, do not advance. Keep `current_stage` on the last confirmed stage, clear queued transitions, and record the blocker or route-drift reason.
7. If the input is raw requirements, a one-line feature idea, a partially specified request, or any request without a PRD artifact, set or keep `current_stage=requirements_brainstorming`, read `requirements-prd-flow.md`, brainstorm the requirement space, resolve all decision-blocking questions, and generate a PRD before advancing.
8. If brainstorming finds unresolved decision-blocking questions that cannot be answered from context or by low-risk defaults, do not call `flutter-prd-rd-writer`. Record the question ledger and stop with `required_inputs`.
9. If all PRD-blocking questions are resolved or explicitly defaulted, invoke `flutter-prd-rd-writer` to create or update the PRD artifact, then queue `pending_next_stage=prd_ready` until confirmation unless `--auto` is active.
10. If the input is already a PRD, broad RD, or feature brief, use `flutter-prd-rd-writer` first to derive or refresh the global technical baseline, then queue or promote `technical_baseline_ready` instead of jumping directly into visual confirmation.
11. If a global technical baseline exists but the workflow lacks a brainstormed global visual direction, route to `flutter-taste-router` first. Global visual design must be brainstormed before final product direction confirmation.
12. If the shared visual direction has been brainstormed but the final product design direction has not yet been explicitly confirmed with the user, set or keep `current_stage=shared_taste_direction`, record `required_inputs=product_design_final_direction_confirmation`, and stop.
13. If the final product design direction is confirmed after the brainstorming step, record `current_stage=product_direction_confirmed` or queue that stage according to the confirmation gate.
14. Do not enter effect-image generation, module design, executable module document generation, architecture, or code before the global visual design direction has been brainstormed and the final product direction has been confirmed.
15. Before global design freeze, generate exactly one representative light-mode effect image from the confirmed product direction. Choose the page that best represents the product's overall hierarchy, component family, density, and CTA posture unless the user explicitly names another page.
16. Before representative effect-image generation, check `IMAGE_BASE_URL` and `IMAGE_API_KEY`. If either is missing, stop and record that effect-image generation is blocked by missing image generation credentials.
17. The representative effect image must explicitly include the confirmed product design direction, PRD constraints, art direction, taste constraints, visual system, CTA posture, palette direction, typography mood, component family cues, image-treatment posture, and target page purpose when those values exist.
18. Save the representative effect image into the global effect-image directory under `docs/rd/` using the corresponding page name, index its path in the workflow record, set the workflow to wait for user confirmation or revision feedback, and do not generate the remaining page effect images yet.
19. After the representative effect image is confirmed, generate the remaining all-page light-mode effect-image set. Do not generate only a partial remaining set unless the user explicitly narrows scope.
20. Save every remaining generated page effect image into the same global effect-image directory under `docs/rd/` using the corresponding page name, and index each path in the workflow record.
21. All newly generated shared or module-stage effect images must be light-mode visuals. Do not generate dark-mode freeze evidence unless the user explicitly requests a dark-only or dual-mode design cycle.
22. Before any global design freeze or module design freeze, always let `flutter-taste-router` normalize the textual design packet first.
23. If Stitch MCP is available and approved all-page effect images must become the structured source of truth, read `stitch-design-source.md`, first confirm whether the user wants a new Stitch project or an existing Stitch project. For `new`, create the Stitch project and freeze the returned `stitch_project_id`; for `existing`, require the user-provided `stitch_project_id` to be confirmed and frozen. Then run Stitch page design in page-scoped subagents with at most 6 page-design subagents in parallel, allow those subagents to download needed image assets for direct use, merge their receipts into the structured design-source packet, and preserve the original effect images as the visual baseline for human visual inspection.
24. If `stitch_project_mode` is missing or ambiguous, stop with `required_inputs=stitch_project_mode:new_or_existing`; do not call Stitch and do not freeze design source. If `stitch_project_mode=existing` and `stitch_project_id` is missing, ambiguous, or not frozen, stop with `required_inputs=stitch_project_id`; do not call Stitch and do not freeze design source. If `stitch_project_mode=new` but project creation cannot return a frozen id, stop with `required_inputs=stitch_project_creation`.
25. If Stitch MCP is required by the design-source policy but unavailable, record `stitch_status=unavailable`, keep the relevant freeze blocked, and do not silently fall back to raw effect images as the only design source.
26. If approved all-page effect images and Stitch output must become a reusable shared design-source contract with fixed light and dark theme values, use `design-preview-to-global-guidelines`, then record `global_guidelines_frozen` as `pending_next_stage` until the user confirms.
27. Before `module_impl_docs_ready`, treat design freezing as shared/public freeze only. Do not treat it as module-private component freeze.
28. If shared design has a candidate direction, all-page effect images, and Stitch output but no explicit freeze decision, use `flutter-design-freeze-gate` directly.
29. If `flutter-design-freeze-gate` finds hierarchy, task guidance, typography, contrast, CTA clarity, or state coverage still weak, do not route to freeze. Allow one shared revision pass through `flutter-taste-router` plus all-page effect-image regeneration when needed, then stop unless the user explicitly restarts the design cycle.
30. If the shared visual direction has been approved but detailed modules and executable module `impl.md` paths do not exist yet, use `flutter-rd-module-splitter` to create module index rows and implementation-final `impl.md` documents in one pass. This pass combines module boundary generation with module implementation-document generation; it must not create split drafts that require a later refinement node.
31. If multiple feature modules depend on one shared route host, root redirect layer, tab shell, or shell-level state, allow `flutter-rd-module-splitter` to split an `app-shell` or `root-shell` module first instead of burying that responsibility inside feature modules.
32. If a selected module lacks an executable `docs/rd/modules/<module>/<module>.impl.md`, route back to `flutter-rd-module-splitter` for a scope-matched regeneration instead of opening a separate refinement stage.
33. Module implementation preparation consumes the executable `impl.md` produced by `flutter-rd-module-splitter`. Do not require or create `docs/rd/modules/<module>/<module>.ui-ux.md`.
34. When `flutter-rd-module-splitter` produces executable module `impl.md` documents, keep `current_stage` at the last confirmed stage, queue each generated module's `impl_status=implementation_final`, and wait for confirmation before module design freeze. In `--auto` mode, auto-apply those queued updates and continue directly into module freeze for the next dependency-safe module.
35. In module freeze, the frozen Stitch design-source packet is required. If the active module lacks a Stitch packet or source page effect image, stop and route back to Stitch design-source generation or validation.
36. If a complete active-module visual draft, effect-image pack, or implementation-facing design-source packet exists, route to Stitch design-source generation or validation first when Stitch MCP is available and `stitch_project_id` is confirmed and frozen, then route to `flutter-design-freeze-gate` before queueing `module_design_frozen`. Treat the module's high-fidelity visual contract as the first freeze question.
37. During executable module document generation and module freeze preparation, do not generate new module real-device effect images by default. Generate them only when `--perviewer` is explicitly active for the current run.
38. When `--perviewer` is active and module-stage effect images are generated, save them into the global effect-image directory under `docs/rd/`, then write that opt-in plus the generated global effect-image paths into `global-design-guidelines.md` so downstream implementation and human visual inspection know those images were intentional module-stage evidence.
39. If `flutter-design-freeze-gate` finds the active-module design package not yet strong enough to freeze, keep the workflow on the active module, update the Stitch packet, module `impl.md`, and optional visual evidence exactly once, and stop without another automatic freeze pass unless the user explicitly restarts a design cycle.
40. When `flutter-design-freeze-gate` approves the active module design source, first verify its receipt includes `high_fidelity_freeze_status=passed` or an explicit approved reduction. Do not immediately mark it frozen. Queue `pending_next_stage=module_design_frozen`, queue `design_source_status=frozen`, and if `impl.md` references the frozen Stitch design-source packet, also queue `impl_status=landed`. In `--auto` mode, auto-apply that promotion and immediately decide whether the same module still needs implementation-readiness or architecture output before switching modules.
41. If a frozen Stitch design source is about to be consumed by implementation RD or code, use `flutter-design-source-control`.
42. If frozen Stitch output, theme values, component rules, and visual evidence must become Flutter-facing tokens, assets, components, screen architecture, and non-native visual fallback decisions, route to `flutter-uiux-to-architecture`.
43. Before architecture planning, implementation-readiness confirmation, human visual inspection, or code implementation, verify that `platform_identifier` is explicit enough for the real validation surface. Do not treat `platform_baseline` such as `ios_hig` as if it already meant `android_emulator`, `windows_desktop`, or `web_browser`.
44. Do not move a module into `implementing` until `technical_baseline_ready`, `module_impl_docs_ready`, `module_design_frozen`, `impl_rd_ready`, and `bootstrap_code_ready` exist for the module, confirmed maturity is at least `impl_status=landed` and `design_source_status=frozen`, the frozen Stitch design-source packet exists, and the required global public code baseline is already landed.
45. The required global public code baseline before module implementation must include at least:
   - app bootstrap and environment initialization
   - root router or route host plus root redirect policy
   - global dependency injection or provider scope entry
   - local storage baseline and persistence wiring
   - global error mapping and logging baseline
   - shared theme or design-token baseline required by the frozen Stitch design source
   - the shared shell layer when feature modules depend on an `app-shell` or `root-shell`
46. Add network baseline and API client wiring to the global public code baseline only when the project or target modules actually require remote data, API access, upload, sync, or other network capabilities. Do not force network infrastructure into purely local, offline, or static-flow projects.
47. As soon as the shared public baseline is explicit enough, prefer triggering `flutter-init` before feature-module code begins. The best trigger point is after shared design freeze, module split, shared shell or global public baseline clarification, and the first architecture output that defines bootstrap-critical inputs.
48. Bootstrap-critical inputs for that early `flutter-init` trigger include at least the global public code baseline, any required `app-shell` or `root-shell`, shared theme or token baseline, route host or redirect policy, storage baseline, error/logging baseline, and network baseline only when the project actually needs it.
49. Do not wait for every feature module to reach `architecture_ready` before triggering `flutter-init` when the shared baseline above is already clear enough.
50. When `--auto` is active after executable module documents are generated, iterate all target modules in dependency-safe order until every module reaches implementation-ready maturity, but allow `flutter-init` to run as soon as the shared bootstrap-critical baseline is ready.
51. In `--auto` mode, after one module reaches any local stable node such as `implementation_final`, `module_design_frozen`, `impl_rd_ready`, or `architecture_ready`, immediately choose the next valid pre-implementation action. That may mean continuing the same module, switching `current_module` to the next dependency-safe module, or triggering `flutter-init` if the shared baseline is ready and the scaffold is still missing.
52. In `--auto` mode, after a module becomes locally complete for the current step, immediately update `current_module`, `current_stage`, `next_skill`, `module_status_table`, and `decision_log` to reflect the next remaining work. Do not leave `next_skill` as a passive future suggestion if unresolved target modules still exist.
53. If a module dependency prevents the next module from advancing safely, keep the workflow active but stop auto-advancement and record the blocker explicitly.
54. If the shared bootstrap-critical baseline is ready and the target project has not been scaffolded yet or does not contain project-local `skills/flutter-dev/`, use `flutter-init` before feature-module implementation begins. Do not delay this just because unrelated feature modules still lack later-stage architecture output.
55. `flutter-init` must stop at initialization boundaries. It may create the directory skeleton and project-local `skills/flutter-dev/` container, but it must not implement bootstrap code, shared wiring, feature code, page code, or module-specific business logic.
56. If `flutter-init` has completed and the directory skeleton plus project-local `skills/flutter-dev/` exist, record `project_initialized` as `pending_next_stage`.
57. After `project_initialized` is confirmed, route to the separate bootstrap code stage. That stage owns global code such as app entry, app shell, shared startup wiring, router host, dependency-injection entry, storage baseline, logging baseline, and other global public code required before feature-module implementation.
58. `project_initialized` is directory-only. Do not treat it as proof that bootstrap code, shared wiring, or any feature implementation already exists.
59. If bootstrap code has landed and the required global public baseline is actually present on disk, record or queue `bootstrap_code_ready`.
60. If `--auto` is active, do not stop for `project_initialized`, `bootstrap_code_ready`, `implementation_final`, `module_design_frozen`, `impl_rd_ready`, `architecture_ready`, or other downstream confirmation gates. Keep advancing until the implementation boundary is reached for all target modules or a blocker appears.
61. If `--auto` is active and all target modules are implementation-ready, stop here instead of entering `implementing`.
62. If implementation work should begin, first route through `@superpowers` to produce `Spec` for the active module. Do not begin execution before that `Spec` exists.
63. After `Spec` is confirmed, route through `@superpowers` again to produce `Plan` for the active module. Do not begin code execution before that `Plan` exists.
64. Only after both `Spec` and `Plan` exist may implementation execution begin through explicit `@superpowers` together with project-local `flutter-dev` and `flutter-project-guardrails`.
65. Implementation execution defaults to parallel execution of independent work packages defined by `Plan`. The orchestrator should split the active module into non-conflicting ownership units and dispatch them in parallel unless shared-file or shared-state conflicts make serial execution necessary.
66. During module implementation, at minimum consider splitting execution into design-source display fidelity and implementation behavior tracks when the work can be owned independently. Both tracks must still be executed through explicit `@superpowers` invocation, not by directly running downstream implementation skills on their own.
67. The default module landing order is: define the minimum data contract first, then land the display layer skeleton and main user path from the frozen Stitch design source, then connect the real data layer.
68. For ordinary business-page modules, prefer `minimum data contract -> display layer -> data integration`. For infrastructure-heavy modules such as auth guards, route shells, sync engines, caches, payment orchestration, or backend-first flows, it is allowed to land the minimum process or data layer first and then attach display surfaces later.
69. Before display-layer code begins, run a display-layer readiness preflight. At minimum, verify:
   - the page has a readable main effect image
   - complex areas have readable detail effect images when needed
   - fidelity-critical regions have enough evidence coverage, including state, scroll-position, and overlay evidence when those dimensions materially affect the layout
   - the frozen Stitch design source explicitly records scroll, list, overlay, layout, sticky, and component-repeatability semantics
   - `flutter-uiux-to-architecture` has already produced a concrete display-layer decision table
   - every important page region is classified as `preserve_faithfully`, `flutterize`, or `simplify`
   - every fidelity-critical region has an explicit `layout_anchor`, `spacing_lock_rule`, `z_axis_rule`, and `pixel_tolerance`
   - non-native visual effects are already classified as native code or asset fallback
   - downloaded image assets required by the Stitch packet have local asset paths and intended regions/components
70. Treat hero regions, CTA clusters, branded illustration zones, layered backgrounds, sticky headers, and overlay-heavy regions as fidelity-critical by default unless the frozen Stitch design source explicitly says otherwise.
71. During display-layer implementation, keep taste guidance active as a guardrail for hierarchy, spacing, typography, contrast, CTA salience, motion restraint, and anti-template composition. Taste must not override frozen Stitch intent. If corresponding page images exist, inspect them through `$image-to-code` before landing display-layer code.
72. Treat effect images as visual-structure evidence, not as the only source of truth for Flutter implementation choices. Final scroll, list, sticky, overlay, and layout decisions must follow the combination of the frozen Stitch design source, `impl.md`, and `flutter-uiux-to-architecture`.
73. If the architecture output marks a region `preserve_faithfully`, do not normalize away its locked spacing, hierarchy, layering, or asset usage for engineering convenience. Any such change must return to `flutter-design-source-control`.
74. If the page effect image or Stitch packet contains a bitmap visual, texture, illustration, photo, composite, or other image asset that can be downloaded from the approved source, download it and let implementation consume the local asset directly. Use `$imagegen` only when the source image is missing, unusable, or explicitly needs generation/rework.
75. Only use `$imagegen` for visuals that are genuinely better as raster assets. Do not use it for shapes, simple gradients, icons that belong to an existing vector system, or effects that Flutter can reproduce cleanly with native code.
76. If code is complete or screenshots exist, stop and request human visual inspection against the frozen Stitch design source and approved page effect images. Do not create a separate automatic review workflow stage and do not route to an automatic visual reviewer by default.
77. If the user requests UI, layout, interaction, hierarchy, visual token, or state changes after shared freeze or module design freeze, use `flutter-design-source-control`.
78. Only route to Pen/Pencil skills when the user explicitly requests Pencil tooling or provides a `.pen` workflow. That optional path must not become a default gate for Flutter implementation.
