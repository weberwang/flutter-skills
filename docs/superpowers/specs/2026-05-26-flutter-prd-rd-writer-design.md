# flutter-prd-rd-writer 设计文档

## 1. 背景

当前仓库已经有面向 Flutter UI 方向的 skill，但缺少一个能够把 PRD 直接转化为完整研发文档的 skill。目标不是生成零散建议，而是生成可交付、可用于研发评审和项目启动的书面方案，覆盖 Flutter 客户端、前后端协作以及产品全链路交付视角。

## 2. 目标

新建一个名为 `flutter-prd-rd-writer` 的 skill，使其能够根据 PRD 产出完整研发文档，并默认覆盖以下能力：

- Flutter 客户端完整技术方案
- 第三方包选型与替代方案
- 第三方包最佳搭档组合与避免混搭建议
- 前后端接口协作边界
- 埋点、监控、风控、灰度、发布等全链路交付方案

## 3. 非目标

- 不直接生成业务代码
- 不直接替代接口文档、UI 设计稿或测试用例明细
- 不在 PRD 信息不完整时伪造细节
- 不为了兼容旧方案而主动引入过渡性实现

## 4. skill 定位

### 4.1 名称与显示信息

- skill 名称：`flutter-prd-rd-writer`
- 显示名称：`Flutter PRD RD Writer`

### 4.2 触发范围

当用户出现以下诉求时应优先触发该 skill：

- 根据 PRD 生成研发文档、技术方案、实施文档、开发设计文档
- 为 Flutter 项目输出技术选型、架构设计、模块拆分、研发落地方案
- 从需求文档直接推导完整 Flutter 端方案、第三方包组合方案、前后端协作边界和全链路交付要求

### 4.3 产物要求

输出必须是完整研发文档，而不是泛泛建议。每个关键技术决策都应说明：

- 为什么这么选
- 不选什么
- 这样选择的代价是什么

## 5. 总体设计

采用“主 skill + 参考手册 + 轻模板”的组织方式：

- `SKILL.md` 负责触发条件、主工作流、输出原则和参考文件分流
- `references/` 负责 PRD 拆解、Flutter 技术方案骨架、第三方包组合策略、全链路交付检查项
- 不引入脚本，先确保 skill 的推理质量和文档结构稳定

这种设计兼顾以下目标：

- 主文件保持轻量，避免上下文膨胀
- 易于持续补充第三方包选型内容
- 允许不同 PRD 在同一文档框架下得到差异化结论

### 6.1 文件职责

- 何时使用该 skill
- 触发后的执行顺序
- 何时读取哪个参考文件
- 输出研发文档时必须遵守的规则

#### agents/openai.yaml

负责定义：

- 显示名称
- 简短描述
- 默认调用提示词

#### references/prd-intake-and-gap-analysis.md

负责定义：

- PRD 信息拆解方法

- 缺失信息识别规则
- 假设前提与待确认项输出格式

#### references/flutter-technical-solution-template.md

负责定义：

- Flutter 研发文档主体章节骨架

#### references/package-selection-and-best-bundles.md

负责定义：

#### references/fullchain-delivery-checklist.md

负责定义：

- 前后端协作要求
- 埋点、监控、风控、灰度、发布、告警、验收等全链路补充内容

## 7. 核心工作流

skill 触发后按以下顺序工作：

1. 读取 PRD，抽取目标、用户、场景、流程、范围与边界
2. 判断 PRD 信息是否足够支持研发方案编写
3. 若信息不足，先输出假设前提与待确认项，再继续研发文档
4. 结合 Flutter 技术方案模板生成主体章节
5. 按能力域生成第三方包选型、最佳搭档和替代方案
6. 补齐前后端协作、数据安全、埋点监控、灰度发布等全链路内容
7. 在结尾汇总推荐主方案、核心风险和优先确认事项

## 8. 默认输出文档框架

### 8.1 文档章节

默认研发文档应包含以下章节：

0. 文档信息
1. PRD 需求理解
2. 假设前提与待确认项
3. 业务流程与页面范围
4. 总体技术架构
5. Flutter 客户端详细技术方案
6. 第三方包选型与最佳搭档
7. 后端协作与接口契约要求
8. 数据与安全方案
9. 埋点、监控与运营支持
10. 测试与质量保障
11. 实施计划与里程碑
12. 风险清单与替代方案
13. 最终结论

### 8.2 强制输出规则

每次输出研发文档时必须遵守：

- 每个关键章节都要包含推荐方案、备选方案和取舍理由
- 每个第三方包建议都要说明适用场景、最佳搭档和避免混搭
- 当 PRD 信息不足时，先输出假设前提与待确认项，不能直接编造细节

## 9. 第三方包最佳搭档策略

### 9.1 选型目标

skill 不输出零散包推荐，而是输出“能力域方案栈”。选型优先级如下：

- 稳定性
- 生态协同
- 维护活跃度
- 团队接入成本

对于版本敏感或生态变化快的包，最终写入研发文档前应基于官方文档或当前包主页进行核实。

### 9.2 选型过程

先判断项目类型：

- 快速 MVP
- 增长型产品
- 复杂业务或企业型产品

再按能力域生成方案，而不是按个人喜好列包。

### 9.3 必须覆盖的能力域

- 状态管理与依赖组织
- 路由与页面守卫
- 网络请求、模型序列化、错误映射
- 本地存储、安全存储、缓存
- 登录态、鉴权、刷新与会话管理
- 图片、媒体、上传下载
- 消息推送、实时通信、深链
- 日志、崩溃、性能监控
- 埋点、远程配置、AB、灰度能力
- 测试、Mock、代码生成、Lint 与工程工具链

### 9.4 输出格式

每个能力域建议统一采用以下结构：

```text
能力域：
推荐主方案：
推荐理由：
最佳搭档：
备选方案：
避免混搭：
接入注意事项：
升级/维护风险：
```

## 10. 质量约束

- `SKILL.md` 保持简洁，只放核心流程与分流指引
- 参考文件聚焦单一职责，避免一个文件混入多个主题
- 包选型文件聚焦规则和组合，不堆砌“包大全”
- 研发文档强调可交付性，而不是讨论式输出

## 11. 实施计划

### 阶段一

初始化 skill 目录，生成基础文件和 `agents/openai.yaml`。

### 阶段二

编写 `SKILL.md`，固化触发条件、主工作流和引用关系。

### 阶段三

补齐 `references/` 内四个参考文件，覆盖 PRD 拆解、技术方案骨架、包选型策略、全链路交付清单。

### 阶段四

运行校验脚本，检查命名、frontmatter、结构完整性。

## 12. 风险与对策

### 风险一：模板过重，限制泛化能力

对策：采用轻模板，只规定输出框架和决策维度，不预填固定内容。

### 风险二：包选型内容过时

对策：在 skill 中明确要求对敏感包先核实现状，再写入最终文档。

### 风险三：输出过于客户端视角

对策：单独维护全链路补充清单，强制覆盖协作、监控、灰度和发布章节。

## 13. 验收标准

当用户给出一份中等复杂度 PRD 时，skill 应能稳定输出：

- 结构完整的研发文档
- 明确的假设前提与待确认项
- 覆盖 Flutter 端与全链路的技术方案
- 成体系的第三方包最佳搭档建议
- 清晰的主方案、备选方案、风险和实施路径
